# Part 1. 프로젝트 개요

---

## 1.1 프로젝트 배경 및 동기

### 마크다운 에디터 시장의 빈틈

마크다운은 개발자, 연구자, 작가 사이에서 사실상의 텍스트 작성 표준이 되었다. 그러나 2025년 현재, 마크다운 에디터 시장에는 해결되지 않은 세 가지 구조적 빈틈이 존재한다.

**첫째, WYSIWYG와 확장성의 양립 부재.** Typora는 "구문이 사라지는" 아름다운 WYSIWYG 경험을 제공하지만 플러그인 시스템이 없고, 클로즈드 소스이며, AI 기능이 전무하다. Obsidian은 1,000개 이상의 커뮤니티 플러그인과 양방향 링크를 제공하지만, Live Preview 방식이라 마크다운 구문이 완전히 사라지지 않는다. 두 에디터의 장점을 하나로 합친 에디터가 없다.

**둘째, LLM 시대의 새로운 편집 니즈 미충족.** Claude Code Skills, ChatGPT Custom Instructions 등 LLM 프롬프트가 `.md` 파일로 작성되는 시대가 되었다. 이 파일들은 YAML frontmatter, 구조화된 섹션, XML 태그, 변수 플레이스홀더(`{{variable}}`)가 혼합된 독특한 형식을 가진다. 기존 에디터는 이런 Skills 파일을 구조적으로 편집하거나, 작성 중인 프롬프트를 즉시 테스트하거나, AI가 프롬프트 품질을 분석해주는 기능을 제공하지 않는다.

**셋째, Notion 사용자의 마크다운 이탈 문제.** Notion에서 KaTeX 수식이 포함된 문서를 마크다운으로 내보내면 수식 구문이 깨지거나 독자 마크업이 포함된다. 연구자와 학생들은 Notion의 협업 환경에서 작업하되, 로컬 마크다운으로도 완벽하게 호환되길 원한다. 현재 이 전환을 매끄럽게 처리하는 에디터가 없다.

### Typora 유료화 이후의 공백

2021년 Typora가 유료화($15)되면서, 오픈소스 대안을 찾는 수요가 급증했다. Mark Text가 대안으로 부상했으나 개발 속도가 느리고 안정성 이슈가 있었다. Markditor(Tauri 기반)는 초기 단계이며, 기능이 제한적이다. "Typora 수준의 WYSIWYG + 오픈소스 + 현대적 기능"이라는 조합은 여전히 빈자리로 남아 있다.

### 에디터에 스며드는 AI

2023~2025년 사이, 에디터와 AI의 통합이 급격히 진화했다. Cursor는 코드 에디터에 Ghost Text 제안, 인라인 Cmd+K 편집, Agent Mode를 도입했다. Notion 3.0은 자율 실행 AI Agent를 출시했다. 이 흐름은 코드 에디터에만 한정되지 않는다. 글쓰기 에디터에서도 AI가 다음 문장을 제안하고, 선택 영역을 번역하고, 문서 전체에 대해 질문에 답하는 기능이 기대되고 있다. 그러나 마크다운 WYSIWYG 에디터 중 이 수준의 AI 통합을 제공하는 에디터는 아직 없다.

## 1.2 프로젝트 비전 및 목표

### 한 줄 비전

> **Baram(바람)은 Typora의 아름다움, Obsidian의 확장성, AI 네이티브 편집을 하나로 합친 오픈소스 WYSIWYG 마크다운 에디터이다.**

### 프로젝트명

**Baram (바람)**

바람은 가볍고, 빠르고, 어디에나 닿는다. 에디터가 지향하는 세 가지 핵심 가치 — 경량성(가볍다), 속도(빠르다), 연결성(어디든 닿는다) — 를 자연스럽게 담고 있다. 영문으로도 부르기 쉽고, 고유한 이름이라 검색과 브랜딩에 유리하다.

### 핵심 가치 (3+2)

세 가지 핵심 가치와 두 가지 보조 가치로 프로젝트의 방향을 정의한다.

| 가치 | 한 줄 정의 | 측정 기준 |
|------|----------|----------|
| 🌬️ **가볍다** | 설치부터 실행까지, 무거움을 느끼지 않는다 | 앱 크기 ~10MB, 메모리 30~50MB, 시작 시간 0.4초 이하 |
| 🎨 **아름답다** | 마크다운 구문이 사라지는, Typora 수준의 WYSIWYG | 포커스 아웃 시 모든 구문 100% 렌더링 |
| 🔗 **연결된다** | 문서, 도구, AI가 자연스럽게 연결된다 | 양방향 링크, Notion 호환, LLM 인라인 통합 |
| 🔓 **열려있다** | 오픈소스이며, 누구나 확장할 수 있다 | MIT 라이선스, 플러그인 API 공개 |
| ⚡ **빠르다** | 타이핑 즉시 렌더링, 지연 없는 편집 | 키 입력 → 렌더링 16ms 이하 (60fps) |

이 중 앞의 세 가지(가볍다, 아름답다, 연결된다)가 Baram의 핵심 정체성이며, 모든 설계 의사결정에서 우선순위 판단 기준으로 사용한다.

### 차별화 전략

기존 에디터와의 포지셔닝을 명확히 한다.

```
              확장성 높음
                 ↑
                 │    ◉ Obsidian
                 │         (플러그인 풍부, WYSIWYG 부족)
                 │
         ◉ Baram│
    (WYSIWYG + 확장성│+ AI 네이티브)
                 │
 WYSIWYG ←───────┼──────→ 소스 편집
                 │
     ◉ Typora   │    ◉ VS Code + Cursor
  (WYSIWYG 최고, │  (AI 최강, 마크다운 에디터 아님)
   확장성 없음,  │
   AI 없음)     │         ◉ Logseq
                 │          (아웃라이너, WYSIWYG 아님)
                 ↓
              확장성 낮음
```

Baram이 목표하는 포지션은 네 영역의 교집합이다.

| 영역 | 참조 대상 | Baram이 가져오는 것 |
|------|----------|-------------------|
| WYSIWYG 품질 | Typora | "구문이 사라지는" 편집 경험, 5대 UX 원칙 |
| 확장성 | Obsidian | Extension-First 아키텍처, 양방향 링크, 커맨드 팔레트 |
| AI 통합 | Cursor, Notion 3.0 | Ghost Text, 인라인 AI 편집, Chat Panel, Agent Mode |
| 경량 + 오픈소스 | Tauri 생태계 | ~10MB, Rust 백엔드, MIT 라이선스 |

기존 에디터가 각각 한두 영역에만 강한 반면, Baram은 네 영역 모두에서 경쟁력 있는 수준을 목표한다.

### 프로젝트 목표 (구체적)

| 목표 | Phase 1 (MVP) | Phase 2 | Phase 3 |
|------|-------------|---------|---------|
| **WYSIWYG 품질** | Typora 핵심 인터랙션 90% 재현 | 테이블/이미지/다이어그램 완성 | Split View, 프레젠테이션 모드 |
| **확장성** | 내부 Extension 패턴 적용 | 양방향 링크 + 백링크 + 블록 참조 | 플러그인 마켓플레이스, Canvas, 그래프 뷰 |
| **AI 통합** | 인라인 AI 편집 + diff + 슬래시 AI 커맨드 | Ghost Text + Chat Panel + Skills AI | Agent Mode + Knowledge Q&A |
| **성능** | 시작 0.4초, 메모리 50MB 이하 | 10,000줄 문서 버벅임 없음 | 대규모 Vault (1,000+ 파일) 인덱싱 |

## 1.3 대상 사용자 (Persona)

### Persona 1: LLM/AI 개발자 (1차 타겟)

> **"프롬프트 .md 파일을 구조적으로 편집하고, 바로 테스트하고 싶다."**

| 항목 | 내용 |
|------|------|
| 직업 | AI 엔지니어, 프롬프트 엔지니어, Claude Code Skills 개발자 |
| 도구 | VS Code, Claude Code, ChatGPT, Cursor |
| 핵심 니즈 | YAML frontmatter 비주얼 편집, `<tags>`/`{{변수}}` 구문 강조, 스킬 간 참조 네비게이션, 프롬프트 즉시 테스트 |
| Pain Point | VS Code에서 프롬프트 .md를 편집하면 구조가 보이지 않는다. 프롬프트를 수정할 때마다 별도 도구에 복사해서 테스트해야 한다. 스킬 간 의존관계를 파악하기 어렵다 |
| Baram의 가치 | Skills 편집 모드(§5.8)에서 YAML을 GUI로 편집하고, 프롬프트를 인라인으로 테스트(§6.4)하며, 양방향 링크로 스킬 간 관계를 탐색(§5.6)할 수 있다 |
| 사용 빈도 | 매일, 하루 2~4시간 |

### Persona 2: 마크다운 파워 유저 (2차 타겟)

> **"Typora처럼 아름답고, Obsidian처럼 강력한 에디터가 필요하다."**

| 항목 | 내용 |
|------|------|
| 직업 | 개발자(기술 문서), 테크니컬 라이터, 블로거 |
| 도구 | Typora, Obsidian, VS Code, Notion |
| 핵심 니즈 | 구문이 사라지는 WYSIWYG, 빠른 시작, 코드 블록/다이어그램, 플러그인 확장, Git 통합 |
| Pain Point | Typora는 완벽하지만 확장이 안 된다. Obsidian은 강력하지만 WYSIWYG가 아니다. 둘 다 AI 기능이 없다 |
| Baram의 가치 | Typora 수준의 편집 경험에 양방향 링크, 플러그인, AI 인라인 편집을 더한다. Tauri 기반이라 가볍다 |
| 사용 빈도 | 매일, 하루 1~3시간 |

### Persona 3: Notion + 마크다운 혼용 사용자 (3차 타겟)

> **"Notion에서 수식 포함 문서를 작성하고, 로컬 마크다운으로도 관리하고 싶다."**

| 항목 | 내용 |
|------|------|
| 직업 | 연구자, 대학원생, 데이터 과학자 |
| 도구 | Notion, Overleaf, Jupyter Notebook, Zotero |
| 핵심 니즈 | KaTeX 수식 완벽 렌더링, Notion 가져오기/내보내기, 참고문헌 관리, PDF 내보내기 |
| Pain Point | Notion에서 내보낸 마크다운의 수식이 깨진다. 로컬 에디터에서 수식을 편집하면 Notion과 호환이 안 된다. 오프라인에서 작업할 수 없다 |
| Baram의 가치 | KaTeX 네이티브 지원(Notion 100% 호환), 양방향 Notion 내보내기/가져오기, 로컬 파일 기반이라 오프라인 완벽 지원 |
| 사용 빈도 | 주 3~5회, 논문/보고서 작성 시 집중 사용 |

### Persona 우선순위 및 설계 영향

```
Persona 1 (LLM 개발자)  ──→  Skills 편집 모드, AI 통합, 블록 참조
         │
         ├── Baram만의 고유 차별화 요소
         │
Persona 2 (MD 파워유저)  ──→  WYSIWYG 코어, 확장성, 성능
         │
         ├── 가장 넓은 사용자층, 에디터의 기본 품질 결정
         │
Persona 3 (Notion+수식)  ──→  KaTeX 호환, Notion 연동
         │
         └── 특정 시장의 강한 니즈, 이탈 유도 포인트
```

Phase 1 설계 시 Persona 2(기본 WYSIWYG 품질)를 최우선으로 만족시키되, Persona 1(Skills/AI)의 핵심 요소를 Phase 1에 포함하여 차별화를 확보한다. Persona 3(수식/Notion)는 KaTeX를 Phase 1에 포함함으로써 일찍 대응한다.

## 1.4 프로젝트 범위 (Scope)

### In-Scope

Baram 설계서가 다루는 범위를 명확히 한다.

**에디터 코어**
- GFM(GitHub Flavored Markdown) + 확장 구문 지원 WYSIWYG 편집기
- "구문이 사라지는" Typora 스타일 실시간 렌더링
- 소스 코드 모드 ↔ WYSIWYG 전환
- 마크다운 파일(.md) 직접 읽기/쓰기 (독자 포맷 없음)

**콘텐츠 블록**
- 테이블, 코드 블록(구문 강조), KaTeX 수식(인라인+블록), Mermaid 다이어그램
- 이미지(드래그&드롭, 클립보드, 리사이즈, 업로드), 콜아웃, 토글

**연결 시스템**
- 양방향 링크 (`[[wikilink]]`), 백링크, 블록 참조 (`((block-id))`), 블록 임베드
- 링크 자동완성, 호버 미리보기, 파일 이름 변경 시 링크 자동 갱신
- 그래프 뷰, 쿼리 블록

**UI/UX**
- 3-Layer Interaction 모델 (Inline / Contextual / Global)
- 커맨드 팔레트, 슬래시 커맨드, 플로팅 서식 툴바
- 파일 트리, 아웃라인, 검색, 북마크 사이드바
- 다크/라이트 테마, 커스텀 CSS 테마

**AI 통합**
- 5-Level AI 통합 (Ghost Text → Inline Edit → Chat Panel → Agent → Knowledge Q&A)
- LLM Provider 추상화 (Claude, OpenAI, Gemini, Local LLM)
- Skills-Aware AI (프롬프트 최적화, 인라인 테스트, 구조 자동 생성)

**플랫폼**
- 데스크톱: macOS, Windows, Linux (Tauri 2.0)
- 파일 형식: 표준 마크다운 (.md) + YAML frontmatter

**내보내기**
- PDF, HTML, Word (기본) + Pandoc 확장 (LaTeX, Epub 등)
- Notion 가져오기/내보내기

### Out-of-Scope

다음은 Baram의 설계 범위에 포함하지 않는다. 일부는 플러그인으로 추후 지원될 수 있다.

| 항목 | 사유 |
|------|------|
| 웹 에디터 버전 | 데스크톱 앱에 집중. 웹 버전은 별도 프로젝트로 검토 |
| 실시간 서버 기반 협업 | Phase 3에서 Yjs 기반 P2P 협업만 포함. 서버 인프라 불포함 |
| 모바일 앱 | Phase 3에서 Tauri Mobile 검토. 초기 설계 범위 밖 |
| 독자 파일 포맷 | 항상 표준 마크다운 출력. 바이너리/독자 포맷 사용 안 함 |
| 데이터베이스 뷰 (Notion 스타일) | Notion의 데이터베이스 기능 복제는 범위 밖. 쿼리 블록으로 일부 대체 |
| PDF 주석 / 논문 관리 | Zotero 등 전문 도구와의 연동은 플러그인으로 추후 고려 |
| 프레젠테이션 모드 | 마크다운 → 슬라이드 변환은 플러그인으로 추후 고려 |
| LLM 모델 학습/파인튜닝 | Baram은 LLM을 호출하는 클라이언트. 모델 자체 개발은 범위 밖 |

### 지원 플랫폼 상세

| 플랫폼 | 지원 시기 | 기술 |
|--------|---------|------|
| **macOS** (Apple Silicon + Intel) | Phase 1 | Tauri 2.0 + WebKit |
| **Windows** (x64 + ARM) | Phase 1 | Tauri 2.0 + WebView2 |
| **Linux** (x64, deb/AppImage) | Phase 1 | Tauri 2.0 + WebKitGTK |
| **모바일** (iOS/Android) | Phase 3+ (검토) | Tauri 2.0 Mobile (실험적) |

macOS를 주 개발 플랫폼으로 사용하되, CI/CD에서 Windows/Linux 빌드를 상시 검증한다. 크로스 플랫폼 WebView 차이(WebKit vs WebView2 vs WebKitGTK)로 인한 렌더링 차이는 기술 리스크로 관리한다(Part 8.3 참조).

## 1.5 설계서 구성 안내

이 설계서는 9개 파트로 구성되며, 각 파트의 역할은 다음과 같다.

| 파트 | 역할 | 읽는 대상 |
|------|------|----------|
| **Part 1. 프로젝트 개요** (본 문서) | 왜 만드는가, 무엇을 만드는가, 누구를 위한 것인가 | 모든 이해관계자 |
| **Part 2. 시장 분석 및 벤치마킹** | 기존 에디터 분석과 설계 방향의 근거 | 제품 기획, 설계자 |
| **Part 3. 아키텍처 설계** | 기술 스택, 시스템 구조, 엔진, 상태 관리 | 개발자 |
| **Part 4. UI/UX 설계** | 3-Layer Interaction 모델, 레이아웃, 메뉴, 테마 | 디자이너, 프론트엔드 개발자 |
| **Part 5. 핵심 기능 상세 설계** | 14개 핵심 기능의 구현 명세 | 개발자 |
| **Part 6. AI 통합 설계** | 5-Level LLM 통합, Skills AI, Provider 추상화 | AI/백엔드 개발자 |
| **Part 7. 데이터 모델 및 파일 규격** | 마크다운 스펙, ProseMirror 스키마, 설정 파일 | 개발자 |
| **Part 8. 개발 로드맵 및 일정** | Phase별 계획, 72개 기능 배정, 마일스톤, 리스크 | 프로젝트 관리자, 모든 이해관계자 |
| **Part 9. 부록** | 용어집, 단축키 맵, 참고 자료 | 참조용 |
