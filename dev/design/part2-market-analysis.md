## 2.1 경쟁 에디터 비교 분석

### 분석 대상 7개 에디터

마크다운 에디터 시장에서 Baram과 직접 경쟁하거나, 설계 참고가 되는 7개 에디터를 분석했다. 각 에디터의 기술 스택, 핵심 강점, 약점, 그리고 Baram 설계에 대한 시사점을 정리한다.

#### Typora — WYSIWYG의 절대 기준

Typora는 마크다운 WYSIWYG 편집의 사실상 표준이다. "구문이 사라지는" 편집 경험은 2015년 출시 이후 어떤 에디터도 완벽히 재현하지 못했다.

<!-- colwidths:119,749 -->

| 항목        | 내용                                                                        |
| --------- | ------------------------------------------------------------------------- |
| 기술 스택     | Electron + Chromium + 자체 엔진                                               |
| 핵심 강점     | 구문 사라짐 WYSIWYG, 즉시 렌더링, 미니멀 UI, KaTeX/Mermaid, 정교한 이미지 관리, Pandoc 기반 내보내기 |
| 약점        | 클로즈드 소스, 플러그인 없음, AI 없음, 협업 없음, Electron 무거움(\~100MB)                     |
| 가격        | \$15 일회성 구매                                                               |
| Baram 시사점 | UI/UX의 절대적 기준점. 5대 UX 원칙(§2.2)과 메뉴/컨텍스트 메뉴/설정 구조를 설계 레퍼런스로 활용             |

#### Obsidian — 확장성과 지식 관리의 최강자

Obsidian은 양방향 링크 기반 지식 관리와 커뮤니티 플러그인 생태계에서 독보적이다. 그러나 WYSIWYG가 아닌 Live Preview 방식이라 마크다운 구문이 완전히 사라지지 않는다.

<!-- colwidths:119,818 -->

| 항목        | 내용                                                                                       |
| --------- | ---------------------------------------------------------------------------------------- |
| 기술 스택     | Electron + CodeMirror 6 + 커스텀 렌더링                                                        |
| 핵심 강점     | 양방향 링크, 2,000+ 플러그인, 커맨드 팔레트, Quick Switcher, Properties 뷰, Canvas, 그래프 뷰, Workspace 프리셋 |
| 약점        | Live Preview(구문 완전히 안 사라짐), 러닝 커브, 앱 크기 \~300MB, MathJax(KaTeX 아님)                       |
| 가격        | 무료 (Sync/Publish 유료)                                                                     |
| Baram 시사점 | Extension-First 아키텍처, 양방향 링크, 커맨드 팔레트, Quick Switcher 확장, 별칭/북마크 시스템의 설계 참조              |

#### Logseq — 블록 레벨 참조와 쿼리의 선구자

Logseq는 아웃라이너 기반으로 문서보다 작은 블록 단위의 참조와 동적 쿼리 시스템이 핵심이다. WYSIWYG 에디터와는 패러다임이 다르지만, 블록 참조와 쿼리 개념은 Baram에 중요한 영감을 제공한다.

<!-- colwidths:119,709 -->

| 항목        | 내용                                                                      |
| --------- | ----------------------------------------------------------------------- |
| 기술 스택     | Electron + ClojureScript + Datascript (DB 버전 전환 중)                      |
| 핵심 강점     | 블록 참조 `(())`, 블록 임베드 `{{embed}}`, 고급 쿼리 시스템, 슬래시 커맨드, 저널/데일리 노트, PDF 주석 |
| 약점        | 아웃라이너(WYSIWYG 아님), 앱 크기 \~300MB, 성능 이슈, DB 전환 진통                        |
| 가격        | 오픈소스 (AGPL)                                                             |
| Baram 시사점 | 블록 참조/임베드 설계, 슬래시 커맨드 패턴, 쿼리 블록 개념, "삽입은 슬래시, 실행은 커맨드" 원칙               |

#### Mark Text — 오픈소스 Typora 대안의 가능성과 한계

Mark Text는 MIT 라이선스의 오픈소스 WYSIWYG 에디터로, Typora의 대안을 목표했다. 그러나 개발 속도가 느리고 안정성 이슈가 있어 실질적인 대안이 되지 못하고 있다.

<!-- colwidths:107,923 -->

| 항목        | 내용                                                                                               |
| --------- | ------------------------------------------------------------------------------------------------ |
| 기술 스택     | Electron + 자체 엔진 (Muya)                                                                          |
| 핵심 강점     | MIT 라이선스, Typora 유사 WYSIWYG, GFM/KaTeX 지원                                                        |
| 약점        | 느린 개발, 안정성 이슈, 고급 기능 부족, 커뮤니티 축소                                                                 |
| Baram 시사점 | 오픈소스 WYSIWYG MD 에디터의 시장 수요 증명. 동시에 자체 엔진의 유지보수 어려움을 보여줌 → Tiptap(ProseMirror) 같은 검증된 프레임워크 선택 근거 |

#### Notion — 수식 호환과 블록 편집의 대중화

Notion은 마크다운 에디터가 아니지만, KaTeX 수식 렌더링과 슬래시 커맨드를 대중화했으며, Baram의 Persona 3(Notion 혼용 사용자)의 기존 도구이다.

<!-- colwidths:119,653 -->

| 항목        | 내용                                                             |
| --------- | -------------------------------------------------------------- |
| 기술 스택     | 독자적 블록 기반 에디터 (React), 2025년 Notion 3.0으로 AI Agent 도입          |
| 핵심 강점     | 블록 기반 편집, KaTeX 수식, 슬래시 커맨드, 실시간 협업, 데이터베이스, AI Agent(3.0)     |
| 약점        | 독자 형식(표준 MD 아님), 내보내기 시 수식 깨짐/UUID 포함, 오프라인 제한                 |
| Baram 시사점 | KaTeX 100% 호환 필수(수식 엔진 동일), 슬래시 커맨드 UX 참고, Notion 가져오기/내보내기 설계 |

#### Nimbalyst — AI 네이티브 편집의 최전선

Nimbalyst는 2025년 등장한 AI 네이티브 마크다운 에디터로, 인라인 AI Diff와 Claude Code 네이티브 통합이 특징이다.

<!-- colwidths:119,798 -->

| 항목        | 내용                                                                       |
| --------- | ------------------------------------------------------------------------ |
| 기술 스택     | 로컬 퍼스트 + AI 네이티브                                                         |
| 핵심 강점     | 인라인 AI Diff, Claude Code 통합, WYSIWYG + 클린 마크다운 출력                        |
| 약점        | 초기 단계, 기능 제한적, 생태계 미성숙                                                   |
| Baram 시사점 | AI 네이티브 마크다운 에디터라는 카테고리가 형성 중임을 증명. Baram은 여기에 Typora급 WYSIWYG 품질을 더해야 함 |

#### Markditor — Tauri + WYSIWYG의 실질적 구현 사례

Markditor는 Tauri 기반 경량 마크다운 에디터로, Baram과 가장 유사한 기술 스택을 가진다.

<!-- colwidths:119,853 -->

| 항목        | 내용                                                                                    |
| --------- | ------------------------------------------------------------------------------------- |
| 기술 스택     | Tauri (preferred) / Electron + React + Vditor + Zustand + Tailwind CSS                |
| 핵심 강점     | Tauri 기반 경량, Vditor의 instant-render 방식                                                |
| 약점        | 초기 단계, 기능 제한적, Vditor의 WYSIWYG 품질이 Typora에 미달                                         |
| Baram 시사점 | Tauri + WYSIWYG 조합의 실현 가능성 확인. 그러나 Vditor 대신 Tiptap을 선택한 이유는 확장성과 WYSIWYG 품질(§3.1 참조) |

## 2.2 Typora 심층 벤치마킹

Typora는 Baram의 WYSIWYG 편집 경험의 직접적 벤치마크이다. Typora의 전체 인터페이스를 전수 분석하여 설계 레퍼런스로 활용한다.

### 전체 레이아웃 구조

Typora는 3-Column 가변 레이아웃을 사용한다.

```
┌─────────────────────────────────────────────────────────────┐
│ [메뉴바]  File  Edit  Paragraph  Format  View  Themes  Help │
├─────────┬───────────────────────────────────────────────────┤
│         │ [타이틀바 / 파일명]                    [🔍][📑][≡] │
│ 사이드바  │                                                   │
│ (토글)   │           메인 에디터 영역                          │
│         │           (WYSIWYG 렌더링)                         │
│ · 파일트리│                                                   │
│ · 파일목록│                                                   │
│ · 아웃라인│                                                   │
│ · 검색   │                                                   │
├─────────┴───────────────────────────────────────────────────┤
│ [상태바]  행:열  |  단어수  |  인코딩  |  줄바꿈  |  테마     │
└─────────────────────────────────────────────────────────────┘
```

핵심 레이아웃 특징: 사이드바는 기본 숨김이며 Ctrl+Shift+L로 토글한다. 에디터 영역에 최대 폭 제한이 있어 읽기 편한 줄 길이를 유지한다. 상태바는 하단에 고정되며 클릭 시 상세 워드카운트 팝업을 제공한다. 반응형으로 동작하여 창 크기 축소 시 사이드바가 자동 숨겨지고, 전체화면(F11) 시 메뉴바까지 숨긴다.

### 메인 메뉴 7개 구조

Typora는 7개 메뉴(File, Edit, Paragraph, Format, View, Themes, Help)를 가진다.

**File 메뉴** — 파일 관리와 내보내기의 허브이다. `Open Quickly`(Ctrl+P)는 VS Code 스타일 퍼지 검색, `Export`는 Pandoc 기반으로 PDF/HTML/Word/LaTeX/Epub 등 10개 이상 포맷을 지원하며 사용자 정의 Export 항목도 추가 가능하다.

**Edit 메뉴** — 기본 편집 외에 `Copy As Markdown`(서식 → MD 변환 복사), `Select Style Scope`(Ctrl+E, 컨텍스트 감응 — 볼드 안이면 볼드 범위, 테이블이면 셀), `Find in Files`(글로벌 검색) 등 마크다운 에디터 특화 기능이 있다.

**Paragraph 메뉴** — 블록 레벨 요소를 제어한다. H1\~H6, 리스트, 인용, 코드 블록, 수식 블록, 테이블(상세 서브메뉴 포함), 수평선, TOC, YAML Frontmatter를 직접 삽입한다. `Increase/Decrease Heading Level`(Ctrl+=/Ctrl+-)로 헤딩 레벨을 빠르게 전환할 수 있다.

**Format 메뉴** — 인라인 레벨 서식을 제어한다. Bold/Italic/Underline, 인라인 코드/수식, Strikethrough/Highlight/Superscript/Subscript, 하이퍼링크, 이미지(경로 관리/업로드 서비스 통합까지 포함된 상세 서브메뉴), Clear Format이 있다. Highlight/Superscript/Subscript는 Preferences에서 활성화해야 사용 가능하다(점진적 복잡도 원칙).

**View 메뉴** — `Source Code Mode`(Ctrl+/)로 WYSIWYG ↔ 소스 전환이 단일 키로 가능하다. Focus Mode(F8), Typewriter Mode(F9), 사이드바 3가지 모드 전환, Zoom 기능이 있다.

**Themes 메뉴** — 5개 기본 테마(Github, Gothic, Newsprint, Night, Pixyll)와 사용자 설치 테마를 전환한다. 테마는 단일 CSS 파일이며, 다크/라이트 모드별 별도 테마를 설정할 수 있다.

**Help 메뉴** — Quick Start, Markdown Reference, Shortcut Keys, Custom Themes, Custom Key Binding 등을 안내한다.

### 컨텍스트 메뉴 7가지 유형

Typora의 컨텍스트 메뉴(우클릭)는 커서 위치의 요소 타입에 따라 완전히 다른 메뉴를 표시한다. 이것은 ProseMirror의 노드 타입 감지와 자연스럽게 연결되는 설계이다.

<!-- colwidths:110,920 -->

| 유형          | 핵심 항목                                                                                                               |
| ----------- | ------------------------------------------------------------------------------------------------------------------- |
| **일반 텍스트**  | Cut/Copy/Paste, Paragraph ▸, Format ▸, Insert ▸, Spelling                                                           |
| **이미지**     | Copy Image/Path, Reveal in Finder, Move/Copy/Upload Image, Rename/Delete, Edit in External Editor, Zoom (25%\~200%) |
| **테이블**     | Add/Delete Row/Column, Align Left/Center/Right, Copy/Delete Table                                                   |
| **코드 블록**   | Copy Code Content, Select All in Block, Change Language, Show/Hide Line Numbers                                     |
| **수식 블록**   | Copy Math Source, Copy As Image, Auto Numbering                                                                     |
| **다이어그램**   | Copy Diagram Source, Copy/Save As Image, Delete Diagram, Zoom                                                       |
| **사이드바 파일** | Open, New File/Folder, Rename, Move/Delete, Reveal in Finder, Copy Path                                             |

설계 시사점: 이미지 컨텍스트 메뉴가 특히 정교하다(경로 관리, 업로드, 줌). 모든 유형에 `Copy As Markdown`과 `Copy As HTML`이 포함된다.

### 사이드바 4가지 모드

사이드바는 같은 영역에서 4가지 모드를 전환하는 방식이다(탭 방식이 아닌 모드 전환).

**파일 트리** (Ctrl+Shift+3) — 폴더 계층 구조, 드래그앤드롭 이동, 자연 정렬(1, 2, 10), 외부 변경 감지/자동 새로고침, 하단에 검색/새 폴더/새 파일 버튼.

**파일 목록** (Ctrl+Shift+2) — 파일명 + 수정일 + 본문 첫 줄 미리보기 3줄 표시, 수정일/생성일/이름순 정렬.

**아웃라인** (Ctrl+Shift+1) — H1\~H6을 계층적 트리로 표시, 클릭 시 해당 위치로 점프, 현재 커서 위치 자동 하이라이트, 필터 검색.

**글로벌 검색** (Ctrl+Shift+F) — 현재 폴더 내 전체 파일 검색, 대소문자/정규식/단어 단위 옵션, 파일별 매칭 수와 컨텍스트 미리보기.

### 설정(Preferences) 8개 탭

Typora의 설정은 8개 탭으로 구성되며, 한국어/영어 검색 기능을 제공한다.

<!-- colwidths:121,909 -->

| 탭              | 핵심 설정                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------- |
| **General**    | 앱 시작 동작(새 파일/폴더 복원/커스텀 폴더), 언어, 업데이트, 고급 설정(JSON)                                           |
| **Appearance** | 윈도우 스타일(독립/통합 타이틀바), 테마(다크 모드 별도), 폰트 크기, 상태바, 사이드바 플로팅                                     |
| **Editor**     | 들여쓰기(2/4/8, 탭/스페이스), 줄 끝 문자, 인코딩, 자동 쌍괄호, 맞춤법, 스마트 구두점, 현재 줄 하이라이트                          |
| **Image**      | 이미지 삽입 정책(현재 폴더/assets/커스텀/업로드), 업로더 선택(iPic/PicGo/PicList/커스텀)                             |
| **Markdown**   | 확장 문법 토글(인라인 수식/첨자/하이라이트/다이어그램), 엄격 모드(# 뒤 스페이스 필수 등), 코드 블록 설정, 수식 옵션(자동 넘버링, physics 패키지) |
| **Export**     | 포맷별 설정(PDF 여백/헤더/푸터, HTML 스타일 포함 여부 등), Pandoc 경로                                           |
| **Files**      | 기본 확장자, 자동 저장 간격, 최근 파일 개수, 파일 연결 설정                                                        |
| **\[검색]**      | 설정 항목을 한국어/영어로 검색하여 빠르게 찾기                                                                  |

### WYSIWYG 동작 패턴

Typora의 "구문이 사라지는" WYSIWYG는 다음 패턴으로 동작한다.

**인라인 요소 (볼드, 이탤릭 등)**: `**텍스트**`를 입력하면 `**`가 사라지고 **텍스트**로 렌더링된다. 해당 영역에 커서를 놓으면 `**텍스트**`가 다시 나타나 편집 가능하다. 커서가 빠지면 다시 사라진다.

**블록 요소 (헤딩)**: `# 제목`을 입력하면 `# `이 사라지고 H1 스타일로 렌더링된다. 블록 시작에 커서를 놓으면 `# `이 다시 나타난다.

**코드 블록/수식**: 코드 펜스(\`\`\`)나 수식 블록(\$\$)은 편집 모드와 렌더링 모드가 전환된다. 블록 안에 커서가 있으면 소스 코드가 보이고, 밖으로 나가면 렌더링 결과만 보인다.

**이미지**: `![alt](url)` 입력 즉시 이미지가 렌더링된다. 이미지를 클릭하면 소스를 편집할 수 있는 인라인 입력창이 나타난다.

### 5대 UX 원칙

Typora의 인터페이스에서 추출한 5가지 핵심 UX 원칙이다. Baram은 이 5가지를 기본으로 계승하고, 2가지를 추가한다(§2.5 참조).

<!-- colwidths:60,299,322,349 -->

| # | 원칙                                   | 정의                         | 구현 전략                                 |
| - | ------------------------------------ | -------------------------- | ------------------------------------- |
| 1 | **구문 불가시성** (Syntax Invisibility)    | 포커스 없는 마크다운 구문은 렌더링 결과로 대체 | Tiptap InputRule + NodeView 포커스/블러 전환 |
| 2 | **즉각 반응** (Instant Feedback)         | 타이핑 즉시 렌더링, 지연 없음          | ProseMirror 트랜잭션 최적화, 디바운스 최소화        |
| 3 | **최소 인터페이스** (Minimal Interface)     | 기본 상태에서 에디터 영역만 보임         | 미니멀 기본 레이아웃, 사이드바/메뉴바 토글              |
| 4 | **마크다운 충실도** (Markdown Fidelity)     | 출력은 항상 클린한 표준 마크다운         | 커스텀 Serializer (ProseMirror → MD)     |

### Typora의 강점과 약점

**배울 점**:

**개선할 점 (Baram의 기회)**:

## 2.3 Obsidian 기능 벤치마킹

Obsidian은 Baram의 확장성과 연결 시스템의 주요 참조 대상이다. Obsidian의 기능을 도입 우선순위에 따라 분류한다.

### 필수 도입 기능 (Baram 코어에 포함)

**양방향 링크 + 백링크** — Obsidian의 핵심 가치. `[[문서명]]`으로 링크를 만들면 대상 문서에 자동으로 백링크가 생성된다. Baram에서는 Skills 간 참조 관계 자동 추적의 근간이 된다. 자동완성 팝업, 링크 자동 갱신(파일명 변경 시), 언링크드 멘션(링크 없이 문서명이 언급된 곳 탐지)을 포함한다.

**커맨드 팔레트** — `Ctrl+P`로 모든 명령을 퍼지 검색. Baram에서는 `Cmd+K`로 할당하여 Cursor 스타일 Command-First UI의 중심으로 발전시킨다. 플러그인 명령, AI 명령, 설정까지 통합 검색한다.

**Quick Switcher** — `Ctrl+O`로 파일을 퍼지 검색하여 빠르게 전환. Baram에서는 `Cmd+O`로 할당하고, 별칭(alias) 매칭, `#`으로 헤딩 직접 이동, `^`로 블록 ID 이동, 검색 결과 없으면 즉시 파일 생성 기능을 추가한다.

**Extension-First 아키텍처** — Obsidian의 가장 큰 교훈은 코어 기능도 플러그인으로 구현한다는 것이다. Baram은 Phase 1부터 내부적으로 Extension 패턴을 사용하고, Phase 3에서 외부 플러그인 API로 개방한다.

### 권장 도입 기능

**별칭(Aliases)** — YAML frontmatter에 `aliases: [별칭1, 별칭2]`를 정의하면 Quick Switcher와 링크 자동완성에서 별칭으로도 검색된다. Skills 파일에서 다양한 이름으로 참조할 때 유용하다.

**Properties 뷰 (비주얼 메타데이터 편집)** — YAML frontmatter를 GUI 폼으로 편집. Baram의 Skills 편집 모드(§5.8)에서 핵심 기능이 된다. 타입 검증, 자동완성, 날짜 피커 등을 포함한다.

**호버 미리보기 (Page Preview)** — 내부 링크 위에 Ctrl+호버하면 대상 노트의 미리보기 팝업이 표시된다. 컨텍스트 전환 없이 참조 내용을 확인할 수 있다.

**Workspace 프리셋** — 사이드바, 패널, 에디터 레이아웃을 이름 붙여 저장/로드. Baram에서는 Writing(에디터만), Research(에디터 + AI 패널 + 아웃라인), Manage(파일 트리 + 에디터 + 백링크) 3가지 기본 프리셋을 제공한다.

**북마크** — 파일, 폴더, 헤딩, 검색어, 블록을 북마크하여 사이드바에서 빠르게 접근. 폴더 구조로 북마크를 조직한다.

**File Recovery** — 자동 스냅샷 + 복원 UI + Diff 뷰. Baram에서는 로컬 스냅샷 시스템과 통합한다.

### 선택/불필요 기능

**그래프 뷰** — D3.js 기반 문서 간 연결 시각화. Phase 3 플러그인으로 구현. 초기에는 핵심 기능이 아님.

**Canvas** — JSON Canvas 표준 호환 무한 캔버스. Phase 3+ 플러그인으로 구현.

**도입 불필요**: Obsidian Sync(Git으로 대체), Obsidian Publish(Export로 커버), Audio Recorder, Random Note, Web Viewer.

## 2.4 Logseq 기능 벤치마킹

Logseq는 아웃라이너 기반으로 Baram과 패러다임이 다르지만, 블록 참조/임베드, 쿼리 시스템, 슬래시 커맨드 등 핵심 개념을 제공한다.

### 필수 도입 기능

**블록 참조&#x20;**`(())` — 문서보다 작은 블록 단위의 참조. `((블록-ID))`를 입력하면 해당 블록의 내용이 인라인으로 표시된다. Obsidian의 `[[문서]]` 링크를 완성하는 마지막 퍼즐이다. Baram에서는 모든 블록에 자동 ID를 부여하고, `((`로 블록 검색 자동완성 팝업을 표시한다.

**블록 임베드&#x20;**`{{embed}}` — 다른 문서의 블록을 현재 문서에 라이브로 삽입(트랜스클루전). 원본이 변경되면 임베드된 내용도 자동 갱신된다. Skills 간 공통 규칙이나 원칙을 재사용할 때 핵심이다.

**슬래시 커맨드** — `/`를 입력하면 블록 삽입 메뉴가 나타난다. Notion에서 대중화되고 Logseq에서 고도화된 패턴이다. Baram에서는 Typora의 마크다운 구문 즉시 변환과 **공존**시킨다 — 마크다운을 아는 사용자는 `# `으로, 모르는 사용자는 `/heading`으로 같은 결과를 얻는다.

### 권장 도입 기능

**저널/데일리 노트** — 앱 시작 시 오늘 날짜의 노트를 자동 생성. 캘린더 위젯에서 날짜를 클릭하면 해당 날짜의 노트로 이동. `@today` 멘션으로 오늘의 노트를 참조. 일일 작업 기록, 회의록, 아이디어 수집에 유용하다.

**쿼리 블록** — 조건 기반으로 콘텐츠를 동적 수집하는 블록. 예를 들어 "status가 진행중인 모든 태스크"를 실시간으로 모아 보여준다. Baram에서는 Logseq의 텍스트 기반 쿼리를 비주얼 쿼리 빌더로 민주화한다.

**태스크 확장** — GFM 체크리스트를 `[/]`(진행중), `[-]`(취소) 상태로 확장. 기본 `[ ]`/`[x]`에 추가 상태를 제공한다.

### 도입 불필요 기능

<!-- colwidths:336,505 -->

| 기능                          | 사유                                          |
| --------------------------- | ------------------------------------------- |
| 아웃라이너 코어                    | Baram은 WYSIWYG 문서 에디터. 아웃라이너 패러다임은 근본적으로 다름 |
| Org-mode                    | 마크다운 전문 에디터. 범위 초과                          |
| DB 그래프 (Datascript)         | 파일 기반 아키텍처. 데이터베이스 레이어는 설계가 완전히 다름          |
| Logseq Sync                 | Git + 클라우드 동기화(iCloud/Dropbox)로 대체          |
| 스페이스드 리피티션 / 네임스페이스 / 화이트보드 | Phase 3+ 플러그인으로 검토                          |

### Logseq에서 배울 핵심 3가지

첫째, **"블록은 1급 시민이다"** — 문서보다 작은 단위를 참조 가능하게 만드는 힘. Obsidian의 `[[문서]]` + Logseq의 `((블록))` = Baram의 3중 연결 시스템.

둘째, **"삽입은 슬래시, 실행은 커맨드"** — `/` = 에디터 내 콘텐츠 삽입(Notion/Logseq 패턴), `Cmd+K` = 앱 전체 명령 실행(Obsidian/VS Code 패턴). 두 진입점의 역할 분리.

셋째, **"쿼리로 지식을 재조합하라"** — 정적 폴더 구조를 넘어 조건 기반으로 콘텐츠를 동적 수집하는 패턴. Skills 관리 대시보드의 기술적 기반.

## 2.5 2025 UI 트렌드 분석

Typora가 출시된 2015년과 현재 2025년 사이, 에디터/생산성 도구의 UI 패러다임이 근본적으로 변화했다. Cursor, Zed, Notion 3.0, Arc 등 최신 앱들에서 관찰되는 트렌드를 분석하고, Baram의 UI 설계 방향에 반영한다.

### Typora 세대 (2015~) vs 현재 세대 (2023~)

<!-- colwidths:85,208,362 -->

| 패턴    | Typora 세대          | 현재 세대 (Zed, Cursor, Notion, Arc)     |
| ----- | ------------------ | ------------------------------------ |
| 메뉴 접근 | 상단 메뉴바 7개 카테고리     | **커맨드 팔레트가 진입점** — 메뉴바는 최소화 또는 숨김    |
| 기능 발견 | 메뉴 순회하며 찾기         | **타이핑으로 검색** — "뭐든 Cmd+K"            |
| 블록 삽입 | 메뉴 또는 단축키 암기       | **슬래시 커맨드&#x20;**`/` — 인라인에서 즉시 선택   |
| 서식 적용 | 메뉴바 Format → 항목 선택 | **플로팅 툴바** — 텍스트 선택 시 바로 위에 팝업       |
| 설정    | 별도 Preferences 창   | **인라인 설정 검색** — 설정도 커맨드 팔레트에서        |
| 레이아웃  | 고정 사이드바 + 메인       | **적응형 패널** — 필요할 때만, 위치 자유           |
| 온보딩   | 매뉴얼/도움말            | **빈 상태(Empty State) 가이드** — 앱 자체가 안내 |
| 테마    | 메뉴에서 선택            | **시스템 테마 자동 연동** + 커맨드 팔레트에서 전환      |
| AI 통합 | 없음                 | **인라인 AI 패널** — 에디터 흐름 안에서 자연스럽게     |

### 핵심 트렌드 4가지

**1) Command-First UI**

Zed의 공식 문서가 첫 줄에서 말한다 — *"The command palette (Cmd+Shift+P) is your gateway to every action."* 메뉴바가 아니라 **커맨드 팔레트가 앱의 중심**이다. 사용자가 "무엇을 하고 싶은지" 타이핑하면 된다. Cursor는 여기에 더해 `Cmd+K`를 AI 인라인 편집의 진입점으로 사용한다.

Baram에의 적용: `Cmd+K`를 커맨드 팔레트의 단축키로 채택. 모든 명령(서식, 내보내기, 테마 전환, AI 명령, 설정, 플러그인)을 하나의 진입점에서 퍼지 검색.

**2) Contextual Floating UI**

Notion이 대중화한 패턴 — 텍스트를 선택하면 **그 자리에** 서식 툴바가 뜨고, 빈 줄에서 `/`를 누르면 **그 자리에** 블록 메뉴가 뜬다. 화면 상단까지 마우스를 올릴 필요가 없다.

Baram에의 적용: 3-Layer Interaction의 Layer 2(Contextual)로 구현. 플로팅 서식 툴바(AI✨ 버튼 포함), 블록 핸들 메뉴(⋮), 테이블/이미지 호버 조작 UI.

**3) Progressive Disclosure with Minimal Chrome**

기본 상태에서 에디터 영역만 보이고, UI 크롬(메뉴바, 사이드바, 상태바)은 최소화한다. 필요한 것은 단축키나 제스처로 즉시 호출한다. Apple의 최신 디자인 언어처럼 "있어야 할 때만 존재"한다.

Baram에의 적용: 메뉴바는 macOS 시스템 메뉴바에 통합(Windows/Linux에서는 최소화), 사이드바 기본 숨김, 상태바 미니멀(워드카운트 + Git 브랜치만).

**4) AI-Native Interaction**

Cursor의 Ghost Text(회색 텍스트로 다음 코드 제안, Tab으로 수락), Cmd+K 인라인 편집(선택 → 자연어 지시 → diff 미리보기), Agent Mode(목표만 주면 여러 파일 자동 편집). Notion 3.0의 자율 AI Agent(20분 자율 실행, 멀티페이지 작업). 이것이 2025년 에디터의 최전선이다.

Baram에의 적용: 5-Level AI 통합 아키텍처(Part 6)로 체계화. 코드 에디터에서 검증된 패턴을 글쓰기 에디터에 적응.

## 2.6 벤치마킹 종합 시사점

### 에디터별 Baram 설계에 대한 기여

```
Typora ───────────→ WYSIWYG 품질, 5대 UX 원칙, 메뉴/설정/컨텍스트 메뉴 설계
                    "구문이 사라지는 경험"의 기준점

Obsidian ─────────→ Extension-First 아키텍처, 양방향 링크, 커맨드 팔레트
                    Quick Switcher, 별칭, Workspace 프리셋

Logseq ───────────→ 블록 참조/임베드, 슬래시 커맨드, 쿼리 블록
                    저널/데일리 노트, "삽입은 슬래시, 실행은 커맨드"

Notion ───────────→ KaTeX 수식 호환, 슬래시 커맨드 UX, 가져오기/내보내기
                    AI Agent 개념 (3.0)

Cursor ───────────→ Ghost Text, Inline Edit (Cmd+K), Chat Panel
                    Agent Mode, diff 리뷰, Privacy Mode, 멀티 모델

Zed ──────────────→ 경량 에디터 철학, 네이티브 성능 우선

Arc ──────────────→ Command-First 네비게이션, 미니멀 크롬
```

### Baram의 7대 설계 원칙 (Typora 5대 + 2025 추가 2대)

Typora에서 계승한 5가지 원칙에 2025 트렌드에서 도출한 2가지를 추가하여, Baram의 7대 설계 원칙을 확정한다.

<!-- colwidths:60,198,147,342 -->

| # | 원칙                         | 출처               | 정의                             |
| - | -------------------------- | ---------------- | ------------------------------ |
| 1 | **구문 불가시성**                | Typora           | 포커스 없는 마크다운 구문은 렌더링 결과로 대체     |
| 2 | **즉각 반응**                  | Typora           | 타이핑 즉시 렌더링, 키 입력 → 렌더링 16ms 이하 |
| 3 | **최소 인터페이스**               | Typora           | 기본 상태에서 에디터 영역만 보임             |
| 4 | **마크다운 충실도**               | Typora           | 출력은 항상 클린한 표준 마크다운 (.md 파일)    |
| 5 | **점진적 복잡도**                | Typora           | 기본 기능은 즉시, 고급 기능은 설정에서 활성화     |
| 6 | **Command-First**          | Cursor, Zed, Arc | 커맨드 팔레트가 모든 기능의 단일 진입점         |
| 7 | **Contextual Floating UI** | Notion, Cursor   | 필요한 UI가 맥락에 맞는 위치에 자동으로 나타남    |

### 기능 통합 계획 요약

벤치마킹 결과를 Phase별로 정리하면 다음과 같다.

**Phase 1 (MVP)에 반영**:
Typora의 WYSIWYG 코어 + 5대 UX 원칙, 커맨드 팔레트(Obsidian), 슬래시 커맨드(Logseq/Notion), 플로팅 서식 툴바(Notion), KaTeX 수식(Notion 호환), 인라인 AI 편집 + diff(Cursor), 블록 핸들 메뉴, Source Code Mode 토글.

**Phase 2에 반영**:
양방향 링크 + 백링크 + 언링크드 멘션(Obsidian), 블록 참조 + 임베드(Logseq), Quick Switcher 확장 + 별칭(Obsidian), 호버 미리보기(Obsidian), 저널/데일리 노트(Logseq), Ghost Text + Chat Panel(Cursor), Skills 편집 모드(Baram 고유), Workspace 프리셋(Obsidian), 북마크(Obsidian).

**Phase 3에 반영**:
Agent Mode + Knowledge Q\&A(Cursor/Notion 3.0), 그래프 뷰(Obsidian), Canvas(Obsidian), 플러그인 시스템(Obsidian), 실시간 협업(Yjs), 쿼리 블록(Logseq), MCP 서버 연동.

### 결론: Baram이 채워야 할 빈자리

현재 시장에는 다음 조합을 제공하는 에디터가 없다.

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │

│                                                             │

│                    + 오픈소스                                 │
│                    + Notion KaTeX 호환                       │
│                    + Skills 편집 특화                         │
│                                                             │
│                    = Baram                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

이 빈자리를 Baram이 채운다.
