# Part 6. AI 통합 설계

---

## 6.1 AI 설계 원칙

Baram의 AI 통합은 5가지 설계 원칙을 따른다. 이 원칙들은 Part 2(§2.5)에서 분석한 2025 에디터 트렌드(Cursor, Notion 3.0, Zed)와 Part 1(§1.2)의 프로젝트 핵심 가치("가볍다 / 아름답다 / 연결된다")로부터 도출되었다.

### 원칙 1: "AI는 보이지 않는 조수"

AI는 UI를 침범하지 않고, 필요할 때만 나타난다. 사용자가 AI를 호출하기 전까지 AI의 존재를 인식할 수 없어야 한다.

```
[좋은 예]
  사용자가 텍스트를 선택 → 플로팅 툴바에 ✨ 버튼 자연스럽게 포함
  사용자가 멈추면 → 회색 Ghost Text가 조용히 나타남
  사용자가 무시하면 → Ghost Text가 조용히 사라짐

[나쁜 예]
  에디터 상단에 "AI 도구 모음" 상시 표시
  AI 제안이 팝업으로 사용자 작업 방해
  "AI를 사용해보세요!" 알림 반복
```

이 원칙은 Part 4(§4.1)의 설계 원칙 3("최소 인터페이스")과 직접 연결된다. AI UI도 3-Layer Interaction 모델을 따라 Layer 1(Ghost Text)과 Layer 2(플로팅 AI 버튼, 인라인 diff)에 자연스럽게 녹아든다.

### 원칙 2: "마크다운 우선"

AI가 생성하는 모든 결과물은 클린한 마크다운으로 출력된다. 독자 포맷이나 특수 태그를 삽입하지 않는다.

```
[AI 출력 규칙]

  1. 서식 적용 → 마크다운 구문 사용 (**볼드**, *이탤릭* 등)
  2. 코드 생성 → 올바른 언어 지정 코드 블록 (```python ... ```)
  3. 수식 생성 → KaTeX 호환 LaTeX ($...$, $$...$$)
  4. 구조 생성 → 표준 마크다운 헤딩, 리스트, 테이블
  5. 메타데이터 → 유효한 YAML frontmatter
```

AI가 삽입한 콘텐츠는 저장 시 일반 마크다운과 구별되지 않는다. Part 1(§1.2)의 "마크다운 충실도" 원칙을 AI에도 동일하게 적용한다.

### 원칙 3: "diff로 신뢰"

AI의 모든 변경 사항은 diff로 미리보기되며, 사용자가 최종 결정한다. AI가 사용자의 텍스트를 즉시 덮어쓰는 일은 없다.

```
[AI 변경 적용 흐름]

  사용자 요청
       │
       ▼
  LLM 호출 (스트리밍)
       │
       ▼
  diff 생성 ──→ 인라인 미리보기 표시
       │         (삽입: 녹색 배경, 삭제: 빨간색 취소선)
       │
       ▼
  사용자 결정
  ├── [✓ 수락]      → 변경 확정, diff 마커 제거
  ├── [✗ 거절]      → 원본 복원
  ├── [↻ 재생성]    → 다른 결과 요청
  └── [✏️ 부분 수정] → diff 행 단위 선택적 수락/거절
```

Agent Mode(Level 4)에서도 이 원칙은 유지된다. 여러 파일을 변경할 때 PR 리뷰 형태의 통합 diff 뷰를 제공한다.

### 원칙 4: "로컬 우선"

API 키가 설정되지 않아도 Baram의 모든 기본 기능은 정상 동작한다. AI는 순수한 부가 기능이다.

```
[AI 비활성 상태]

  API 키 미설정 → AI 관련 UI 자동 숨김
  ├── Ghost Text: 표시 안 됨
  ├── 플로팅 툴바 ✨ 버튼: 숨김
  ├── 슬래시 AI 커맨드: 목록에서 제외
  ├── Chat Panel: 사이드바에서 숨김
  └── 에디터 기본 기능: 100% 정상 동작

[로컬 LLM 지원]

  Ollama / llama.cpp 연동으로 인터넷 없이 AI 사용 가능
  ├── API 키 불필요
  ├── 데이터 외부 전송 없음
  └── Privacy Mode 자동 적용
```

### 원칙 5: "점진적 통합"

AI 기능은 Phase별로 Level 1→5 순서로 구현한다. 각 Level은 독립적으로 동작하며, 이전 Level이 반드시 필요하지 않다.

```
Phase 1 (MVP)  → Level 2: Inline Edit (가장 명시적, 사용자 주도)
Phase 2        → Level 1: Ghost Text + Level 3: Chat Panel
Phase 3        → Level 4: Agent Mode + Level 5: Knowledge Q&A
```

Level 번호는 자율성 수준(낮을수록 암시적)을 나타내며, 구현 순서와 다르다. Level 2(명시적 편집)를 먼저 구현하여 사용자가 AI와의 상호작용에 익숙해진 후, Level 1(암시적 제안)을 추가한다.

## 6.2 5-Level AI 통합 아키텍처

Part 2(§2.5)에서 분석한 Cursor, Notion 3.0, Zed의 AI 패턴을 5단계 아키텍처로 체계화한다. Part 3(§3.4)의 Extension 시스템으로 각 Level은 독립된 Extension으로 구현되어, 개별 활성화/비활성화가 가능하다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    5-Level AI 아키텍처                            │
│                                                                 │
│  Level 5 ┌─────────────────────────┐                           │
│  지식Q&A  │ Vault 전체 지식 검색      │  자율성 ▲               │
│          └──────────┬──────────────┘                           │
│  Level 4 ┌──────────┴──────────────┐                           │
│  Agent   │ 멀티파일 자율 편집        │                           │
│          └──────────┬──────────────┘                           │
│  Level 3 ┌──────────┴──────────────┐                           │
│  Chat    │ 대화형 어시스턴트         │                           │
│          └──────────┬──────────────┘                           │
│  Level 2 ┌──────────┴──────────────┐                           │
│  Inline  │ 선택 → 변환 (diff)       │  ← Phase 1에 구현         │
│          └──────────┬──────────────┘                           │
│  Level 1 ┌──────────┴──────────────┐                           │
│  Ghost   │ 암시적 텍스트 제안        │  자율성 ▼               │
│          └─────────────────────────┘                           │
│                                                                 │
│  ────────────────────────────────────────────────────────────   │
│  공통 기반: LLM Provider 추상화 레이어 (§6.3)                     │
└─────────────────────────────────────────────────────────────────┘
```

### Level 1: Ghost Text (암시적 제안) — Phase 2

타이핑이 잠시 멈추면 AI가 다음에 올 문장/구절을 회색 텍스트로 조용히 제안한다. Cursor의 Tab 자동완성과 GitHub Copilot에서 검증된 패턴이다.

**기본 동작**

```
[Ghost Text 라이프사이클]

  사용자 타이핑 ──→ 디바운스 (500ms 무입력)
       │
       ▼
  컨텍스트 수집 ──→ LLM 호출 (스트리밍)
       │
       ▼
  Ghost Text 표시 (커서 위치에 회색 텍스트)
       │
       ├── Tab      → 전체 수락, 본문에 삽입
       ├── Cmd+→    → 단어 단위 부분 수락
       ├── Esc      → 명시적 무시
       └── 타이핑    → 자동 소멸, 새 제안 준비
```

**컨텍스트 인식 특수 모드**

Ghost Text는 커서가 위치한 블록 타입에 따라 제안 내용을 자동 전환한다.

| 커서 위치 | 제안 내용 | 예시 |
|-----------|-----------|------|
| 일반 텍스트 | 다음 문장/구절 | "주요 특징으로는 WYSIWYG 편집..." |
| 코드 블록 | 코드 완성 | `function parse(input: string)` |
| 수식 블록 | KaTeX 명령어 | `\frac{}{}` → Tab 수락 (§5.3 연동) |
| Skills 파일 | 프롬프트 구조 | `<instructions>` 태그 제안 |
| YAML frontmatter | 속성 값 | `description: "이 스킬은..."` |
| 리스트 항목 | 다음 항목 | `- 세 번째 특징은 성능 최적화...` |

**프롬프트 구성 파이프라인**

Ghost Text의 품질은 LLM에 전달하는 컨텍스트에 의존한다. 토큰 예산 내에서 최대한 관련성 높은 정보를 포함하기 위해 3단계 파이프라인을 사용한다.

```
[프롬프트 구성 — 우선순위 순]

  ┌─ 1순위: 현재 파일 ─────────────────────────────────────────┐
  │  · 커서 앞 텍스트 (최대 2000 토큰)                            │
  │  · 커서 뒤 텍스트 (최대 500 토큰)                             │
  │  · frontmatter 메타데이터 (파일 유형, 태그)                    │
  └──────────────────────────────────────────────────────────────┘
       │
       ▼
  ┌─ 2순위: 열린 파일들 ───────────────────────────────────────┐
  │  · 열린 탭 파일들의 요약 (파일명 + frontmatter + 첫 5줄)      │
  │  · 최근 편집 순 정렬                                          │
  │  · 최대 3개 파일                                              │
  └──────────────────────────────────────────────────────────────┘
       │
       ▼
  ┌─ 3순위: 워크스페이스 컨텍스트 ──────────────────────────────┐
  │  · 현재 파일이 참조하는 [[wikilink]] 대상 파일의 요약           │
  │  · 현재 파일에 대한 백링크 파일 목록                            │
  │  · 프로젝트 루트의 README.md 요약 (있는 경우)                  │
  └──────────────────────────────────────────────────────────────┘
```

**캐싱 및 레이턴시 최적화**

Ghost Text는 타이핑 흐름을 방해하지 않아야 하므로, 레이턴시 최적화가 핵심이다.

| 전략 | 구현 | 효과 |
|------|------|------|
| 디바운스 | 500ms 무입력 후 요청 | 불필요한 API 호출 방지 |
| 스트리밍 | 첫 토큰 도착 즉시 표시 시작 | 체감 응답 시간 단축 |
| 프리페치 | 문단 끝에서 다음 문단 미리 요청 | 연속 작성 시 즉시 표시 |
| 접두사 캐시 | 동일 접두사의 이전 응답 재사용 | 반복 편집 시 API 절약 |
| 요청 취소 | 사용자 타이핑 시 진행 중 요청 즉시 취소 | 낡은 제안 방지 |
| 토큰 제한 | Ghost Text 응답 최대 150 토큰 | 빠른 응답 + 비용 절감 |

**ProseMirror 통합 — Decoration 기반**

Ghost Text는 Part 3(§3.3)에서 정의한 `GhostText` Decoration으로 구현한다. 실제 문서에 삽입하지 않고, 시각적으로만 표시한다.

```typescript
// Ghost Text Decoration 생성
const ghostDecoration = Decoration.widget(cursorPos, () => {
  const span = document.createElement('span');
  span.className = 'ghost-text';           // --syntax-ghost (#d1d5db) 색상
  span.textContent = ghostTextContent;
  span.setAttribute('data-ghost', 'true'); // 직렬화에서 제외
  return span;
});

// Tab 키 수락 시: Decoration 제거 + 실제 텍스트 삽입
// Esc / 타이핑 시: Decoration만 제거
```

**설정**

| 설정 항목 | 기본값 | 설명 |
|-----------|--------|------|
| Ghost Text 활성화 | OFF | API 키 설정 시 자동 ON 제안 |
| 디바운스 간격 | 500ms | 100~2000ms 조절 가능 |
| 최대 제안 길이 | 1문장 | 1문장 / 1문단 / 자유 |
| 특수 모드 | 전체 활성 | 코드/수식/Skills 개별 토글 |

### Level 2: Inline Edit (선택 → 변환) — Phase 1

사용자가 명시적으로 AI를 호출하여 텍스트를 변환하는 레벨이다. 3가지 진입점을 제공한다.

**진입점 1: 플로팅 서식 툴바 AI 버튼 (✨)**

텍스트 선택 시 나타나는 플로팅 서식 툴바(Part 4, §4.2.2)에 포함된 AI 버튼이다.

```
┌───────────────────────────────────────────┐
│ B  I  U  S  </>  🔗  ==  ✨AI  ⋯         │
└───────────────────────────────────────────┘
                              │
                              ▼ 클릭 시
                   ┌──────────────────────────┐
                   │ ✏️ 개선 (Improve)         │
                   │ 📏 줄이기 (Shorten)       │
                   │ 📖 늘리기 (Expand)        │
                   │ 🌐 번역 ▸                 │ → 한/영/일/중
                   │ 🎭 톤 변경 ▸              │ → 격식/캐주얼/기술/친근
                   │ 💡 설명 (Explain)         │
                   │ ─────────────────────── │
                   │ ⌨️ 커스텀 지시...          │
                   └──────────────────────────┘
```

각 명령의 동작 흐름은 동일하다: 선택 영역 텍스트 → 시스템 프롬프트 + 사용자 텍스트로 LLM 호출 → 스트리밍 diff 표시 → 수락/거절.

| 명령 | 시스템 프롬프트 핵심 | 출력 |
|------|---------------------|------|
| 개선 | 문법, 명확성, 흐름 개선. 원문 의미 유지 | 수정된 텍스트 |
| 줄이기 | 핵심만 남기고 간결하게. 원문의 50% 이내 | 축약 텍스트 |
| 늘리기 | 세부 사항, 예시, 설명을 추가하여 확장 | 확장 텍스트 |
| 번역 | 대상 언어로 번역. 기술 용어는 원문 병기 | 번역 텍스트 |
| 톤 변경 | 지정된 톤으로 다시 쓰기. 내용은 유지 | 톤 변경 텍스트 |
| 설명 | 선택 텍스트를 쉽게 설명 | 설명 텍스트 (선택 영역 아래에 삽입) |

**진입점 2: Cmd+K 인라인 프롬프트**

텍스트를 선택한 상태에서 Cmd+K를 누르면 인라인 프롬프트 입력 필드가 나타난다. Part 4(§4.2.3)에서 설계한 인라인 AI 편집 패널이다.

```
[Cmd+K 인라인 편집 — 전체 흐름]

  1. 텍스트 선택 + Cmd+K
     ┌──────────────────────────────────────────┐
     │ 선택된 텍스트가 여기에 하이라이트됩니다.     │
     └──────────────────────────────────────────┘
              ↓
     ┌──────────────────────────────────────────┐
     │ 💬 AI에게 지시: [더 간결하게 다시 써줘     ]│
     │                              [Enter ↵]  │
     └──────────────────────────────────────────┘

  2. LLM 호출 중 (스트리밍)
     ┌──────────────────────────────────────────┐
     │ 선택된 텍스트가 여기에 하이라이트됩니다.     │  ← 빨간 취소선
     │ 선택 텍스트가 하이라이트됩니다.              │  ← 녹색 배경 (스트리밍 중)
     └──────────────────────────────────────────┘
              ↓
     ┌──────────────────────────────────────────┐
     │ ⏳ 생성 중...  ██████░░░░                  │
     └──────────────────────────────────────────┘

  3. 결과 diff 미리보기
     ┌──────────────────────────────────────────┐
     │ 선택된 텍스트가 여기에 하이라이트됩니다.     │  ← 빨간 취소선
     │ 선택 텍스트가 하이라이트됩니다.              │  ← 녹색 배경
     └──────────────────────────────────────────┘
              ↓
     ┌──────────────────────────────────────────┐
     │ [✓ 수락]  [✗ 거절]  [↻ 재생성]  [✏️ 수정]  │
     └──────────────────────────────────────────┘
```

**선택 없이 Cmd+K**: 빈 줄이나 커서 위치에서 Cmd+K를 누르면, 새 콘텐츠를 생성하는 모드로 동작한다. "이 주제에 대한 소개 문단을 써줘"처럼 자연어 지시를 입력하면, 커서 위치에 텍스트를 생성하여 삽입한다(diff 없이 바로 삽입, Undo로 취소 가능).

**진입점 3: 슬래시 AI 커맨드**

빈 줄에서 `/ai-` 접두사로 시작하는 슬래시 커맨드이다. Part 5(§5.11)에서 정의한 슬래시 커맨드 시스템의 AI 카테고리에 해당한다.

| 커맨드 | 동작 | 입력 |
|--------|------|------|
| `/ai-write` | 주제를 받아 초안 생성 | 주제/지시 텍스트 |
| `/ai-brainstorm` | 아이디어 리스트 생성 | 주제 |
| `/ai-translate` | 현재 문서 또는 선택 영역 번역 | 대상 언어 |
| `/ai-summarize` | 문서 또는 선택 영역 요약 | (자동: 현재 문서) |
| `/ai-expand` | 선택 영역 확장 | (자동: 선택 영역) |
| `/ai-fix-grammar` | 문법 교정 | (자동: 현재 문단 또는 선택) |
| `/ai-explain` | 선택 영역에 대한 설명 생성 | (자동: 선택 영역) |

슬래시 AI 커맨드 실행 시 커서 위치 아래에 결과가 스트리밍으로 삽입된다. 삽입된 블록은 녹색 좌측 보더로 "AI 생성 블록"임을 일시적으로 표시하며(5초 후 페이드아웃), Undo(Cmd+Z)로 전체 제거할 수 있다.

**AI Diff Engine 구현**

인라인 편집의 핵심인 diff 엔진은 다음과 같이 동작한다.

```
[AI Diff Engine 파이프라인]

  입력: 선택 영역 텍스트 (original)
    │
    ▼
  LLM 호출 (스트리밍)
    │
    ├── 청크 수신마다:
    │     1. 기존 텍스트와 diff 계산 (문자 수준)
    │     2. ProseMirror Decoration으로 삽입/삭제 마킹
    │     3. 실시간 갱신 (부드러운 전환 애니메이션)
    │
    ▼
  완료 시: diff 결과 확정
    │
    ├── 전체 수락: original 제거, modified 확정
    ├── 전체 거절: modified 제거, original 복원
    ├── 부분 수락: 행(hunk) 단위로 개별 수락/거절
    └── 재생성: modified 제거, 새 LLM 호출
```

diff 시각화 CSS:

```css
.ai-diff-insert {
  background-color: rgba(34, 197, 94, 0.15);   /* 녹색 삽입 */
  text-decoration: none;
}
.ai-diff-delete {
  background-color: rgba(239, 68, 68, 0.15);   /* 빨간색 삭제 */
  text-decoration: line-through;
  opacity: 0.7;
}
.ai-diff-action-bar {
  border: 1px solid var(--border-primary);
  border-radius: 8px;
  backdrop-filter: blur(8px);                   /* 글래스모피즘 */
}
```

**부분 수락 (Hunk-Level Accept)**

diff가 여러 위치에 걸쳐 있을 때, 각 변경 단위(hunk)별로 독립적으로 수락/거절할 수 있다.

```
[부분 수락 UI]

  원문의 첫 번째 문장이다.          ← 변경 없음
  두 번째 문장이 수정되었다.         ← 빨간 취소선      [✓] [✗]
  두 번째 문장을 개선했다.           ← 녹색 배경        ↑ hunk 1
  세 번째 문장은 그대로다.           ← 변경 없음
  네 번째 문장도 수정되었다.         ← 빨간 취소선      [✓] [✗]
  네 번째 문장을 다듬었다.           ← 녹색 배경        ↑ hunk 2
```

각 hunk 오른쪽에 미니 수락/거절 버튼이 표시된다. 모든 hunk에 대한 결정이 완료되면 diff 마커가 제거된다.

### Level 3: Chat Panel (대화형 어시스턴트) — Phase 2

사이드바에 AI 대화 패널을 열어, 문서 맥락 기반의 멀티턴 대화를 수행한다. Part 4(§4.3)에서 설계한 우측 사이드바의 "AI 채팅" 모드이다.

**패널 구조**

```
[AI Chat Panel — Cmd+Shift+A]

  ┌──────────────────────────────┐
  │ 💬 AI Chat          [⚙️] [✕] │
  │ ────────────────────────────│
  │                              │
  │  🤖 안녕하세요! 현재 문서에 대해  │
  │     질문하거나 편집을 요청하세요. │
  │                              │
  │  👤 이 문서의 두 번째 섹션을     │
  │     더 자세하게 확장해줘.        │
  │                              │
  │  🤖 두 번째 섹션을 확장했습니다: │
  │     ...                      │
  │     [📝 에디터에 적용]          │
  │                              │
  │ ────────────────────────────│
  │ [@파일] [+컨텍스트]            │
  │ ┌──────────────────────────┐│
  │ │ 메시지 입력...             ││
  │ │                    [↵]   ││
  │ └──────────────────────────┘│
  └──────────────────────────────┘
```

**@ 참조 시스템**

Chat Panel 입력 필드에서 `@`를 입력하면 컨텍스트 참조 자동완성이 나타난다.

| 참조 | 설명 | 예시 |
|------|------|------|
| `@파일명` | 특정 파일을 대화 컨텍스트에 포함 | `@README.md` |
| `@selection` | 에디터에서 현재 선택된 텍스트 참조 | `@selection을 번역해줘` |
| `@clipboard` | 클립보드 내용 참조 | `@clipboard 내용을 정리해줘` |
| `@current` | 현재 열린 파일 전체 참조 (기본값) | `@current 요약해줘` |
| `@folder/` | 폴더 내 모든 파일 참조 | `@skills/ 분석해줘` |

`@` 입력 후 퍼지 검색으로 파일을 선택할 수 있다. 파일명, 별칭(alias), 경로 모두 검색된다. Part 5(§5.6)의 링크 자동완성과 동일한 검색 인덱스를 공유한다.

**Apply to Editor 동작**

Chat Panel에서 AI가 코드나 텍스트를 제안하면, "에디터에 적용" 버튼이 표시된다.

```
[Apply to Editor 흐름]

  AI 응답에 코드/텍스트 블록 존재
       │
       ▼
  [📝 에디터에 적용] 버튼 클릭
       │
       ▼
  적용 대상 결정
  ├── @selection 참조 있음 → 선택 영역 교체 (diff 표시)
  ├── @current 참조 있음   → 문서 내 관련 위치에 교체/삽입 (diff 표시)
  └── 참조 없음            → 커서 위치에 삽입 (diff 없이, Undo 가능)
       │
       ▼
  diff 미리보기 (Level 2 Inline Edit과 동일한 UI)
       │
       ▼
  [✓ 수락] / [✗ 거절]
```

**대화 히스토리 관리**

| 항목 | 설계 |
|------|------|
| 저장 위치 | `.baram/chat-history/` 디렉토리, 세션별 JSON 파일 |
| 세션 단위 | 파일 변경 또는 명시적 "새 대화" 시 새 세션 생성 |
| 히스토리 표시 | 패널 상단 드롭다운으로 이전 세션 선택 |
| 컨텍스트 윈도우 | 최근 20턴까지 유지, 초과 시 요약 후 앞부분 제거 |
| 삭제 | 개별 세션 삭제 또는 전체 히스토리 초기화 |

### Level 4: Agent Mode (자율 실행) — Phase 3

사용자가 목표만 제시하면 AI가 여러 파일에 걸쳐 자율적으로 편집을 수행한다. Cursor의 Composer/Agent와 Notion 3.0의 자율 AI Agent에서 영감을 받았다.

**Agent Mode 진입**

```
[Agent Mode 진입 방법]

  1. 커맨드 팔레트: "AI: Agent Mode" 실행
  2. Chat Panel에서: "여러 파일에 걸쳐..." 등 멀티파일 의도 감지 시
     → "Agent Mode로 전환할까요?" 제안
  3. 슬래시 커맨드: (없음 — 단일 파일 대상이 아니므로)
```

**실행 흐름**

```
[Agent Mode 실행 파이프라인]

  1. 목표 입력
     "모든 스킬 파일의 description을 검토하고, 누락되거나 모호한 것을 개선해줘"
       │
       ▼
  2. 계획 수립 (AI가 생성, 사용자에게 표시)
     ┌────────────────────────────────────────┐
     │ 📋 실행 계획                             │
     │                                        │
     │  1. skills/ 폴더의 모든 .md 파일 스캔     │
     │  2. frontmatter의 description 추출       │
     │  3. 누락/모호한 description 식별          │
     │  4. 개선된 description 작성              │
     │  5. 각 파일에 변경 적용                   │
     │                                        │
     │  대상 파일: 12개                          │
     │  예상 변경: ~8개 파일                     │
     │                                        │
     │  [▶ 실행] [✏️ 계획 수정] [✗ 취소]         │
     └────────────────────────────────────────┘
       │
       ▼
  3. 자율 실행 (스트리밍 진행 상황 표시)
     ┌────────────────────────────────────────┐
     │ ⏳ 실행 중... (3/12 파일 완료)            │
     │                                        │
     │  ✅ skills/summarizer.md — 변경 없음      │
     │  ✅ skills/translator.md — description 개선│
     │  🔄 skills/analyzer.md — 처리 중...       │
     │  ⏸️ skills/deployer.md — 확인 필요         │
     │  ⬚ skills/formatter.md — 대기 중          │
     │  ...                                    │
     └────────────────────────────────────────┘
       │
       ├── ⏸️ 위험한 변경 감지 시:
       │   "deployer.md의 requires 필드도 변경해야 합니다.
       │    진행할까요?" [✓ 예] [✗ 건너뛰기]
       │
       ▼
  4. 결과 통합 diff 뷰 (PR 리뷰 형태)
     ┌────────────────────────────────────────┐
     │ 📊 Agent 결과 — 8개 파일 변경             │
     │                                        │
     │  📄 skills/translator.md     +3 -1      │
     │  📄 skills/analyzer.md       +5 -2      │
     │  📄 skills/formatter.md      +2 -1      │
     │  ...                                    │
     │                                        │
     │  [파일 클릭 → diff 상세 보기]              │
     │                                        │
     │  [✓ 전체 수락] [✗ 전체 거절] [📝 파일별 선택]│
     └────────────────────────────────────────┘
```

**안전 장치**

Agent Mode는 자율성이 높으므로, 다음 안전 장치를 적용한다.

| 장치 | 설명 |
|------|------|
| 계획 확인 | 실행 전 반드시 사용자가 계획을 확인/수정 |
| 위험 감지 중단 | 구조적 변경(파일 삭제, frontmatter 스키마 변경) 시 중단 후 확인 |
| 실행 중 취소 | 언제든 "중지" 버튼으로 즉시 중단, 이미 적용된 변경은 diff 뷰로 표시 |
| 통합 diff 리뷰 | 모든 변경을 한 화면에서 파일별로 검토 |
| 파일별 수락/거절 | 일부 파일의 변경만 선택적으로 적용 |
| Undo 일괄 지원 | 수락 후에도 Cmd+Z로 Agent 실행 전 상태로 일괄 복원 |

**Agent Mode 활용 시나리오**

| 시나리오 | 사용자 입력 |
|----------|-----------|
| README 자동 생성 | "이 프로젝트의 README를 작성해줘" |
| Skills 일괄 개선 | "모든 스킬의 description을 검토하고 개선해줘" |

| 코드 문서화 | "모든 코드 블록에 주석을 추가해줘" |

  4. 답변 + Citation 표시
     ┌────────────────────────────────────────┐
     │ 🤖 이 프로젝트에서는 JWT 기반 토큰 인증을  │
     │    사용합니다. [1] 인증 미들웨어는         │
     │    auth/middleware.md에 정의되어 있으며,   │
     │    토큰 갱신 로직은 [2]에서 확인할 수       │
     │    있습니다.                              │
     │                                        │
     │    출처:                                 │
     │    [1] auth/middleware.md#JWT-검증        │  ← 클릭 시 이동

     └────────────────────────────────────────┘

**인덱싱 전략**

| 항목 | 설계 |
|------|------|
| 인덱싱 단위 | 문서를 헤딩 기준 청크(chunk)로 분할 |
| 임베딩 모델 | Provider의 embed() 사용 (OpenAI text-embedding-3-small 또는 로컬 모델) |
| 저장 위치 | `.baram/embeddings/` (벡터 인덱스 파일) |
| 갱신 전략 | 파일 저장 시 해당 파일의 청크만 증분 갱신 |
| 링크 그래프 연동 | Part 5(§5.6)의 백링크 인덱스 활용, 관련 문서 범위 확장 |
| 초기 빌드 | Vault 열기 시 백그라운드에서 전체 인덱싱 (진행률 상태바 표시) |

**활용 시나리오**

| 질문 유형 | 예시 |
|-----------|------|
| 개념 검색 | "이 프로젝트의 에러 처리 전략은?" |
| 참조 추적 | "skill-a.md를 참조하는 모든 스킬을 알려줘" |
| 요약 | "이 프로젝트 전체를 요약해줘" |
| 비교 | "v1과 v2의 API 차이점은?" |
| 누락 탐지 | "테스트가 없는 기능은 어떤 것들이 있어?" |

Knowledge Q&A는 Chat Panel(Level 3) 안에서 동작한다. Chat Panel에서 질문이 Vault 전체 검색을 필요로 한다고 판단되면 자동으로 Knowledge Q&A 파이프라인으로 전환된다.

## 6.3 LLM Provider 추상화 레이어

모든 AI Level이 공유하는 Provider 추상화 레이어이다. Part 3(§3.4)의 Extension API에서 `ai.complete()` / `ai.isAvailable()`로 노출되는 인터페이스의 내부 구현이다.

### Provider 인터페이스

```typescript
interface LLMProvider {
  readonly id: string;                    // 'claude' | 'openai' | 'gemini' | 'ollama'
  readonly name: string;                  // 'Claude (Anthropic)'
  readonly models: ModelInfo[];           // 사용 가능한 모델 목록

  // 핵심 메서드
  complete(request: CompletionRequest): Promise<CompletionResponse>;
  stream(request: CompletionRequest): AsyncIterable<StreamChunk>;
  embed(text: string): Promise<number[]>;  // Knowledge Q&A용

  // 상태
  isAvailable(): boolean;                 // API 키 설정 + 연결 확인
  estimateCost(tokens: number): number;   // 예상 비용 (USD)
}

interface CompletionRequest {
  model: string;                          // 'claude-sonnet-4-5-20250514'
  messages: Message[];                    // 시스템 + 사용자 + 어시스턴트
  maxTokens: number;
  temperature: number;
  stopSequences?: string[];
}

interface StreamChunk {
  text: string;                           // 텍스트 청크
  done: boolean;                          // 스트리밍 완료 여부
  usage?: { inputTokens: number; outputTokens: number };
}

interface ModelInfo {
  id: string;                             // 'claude-sonnet-4-5-20250514'
  name: string;                           // 'Claude Sonnet 4.5'
  contextWindow: number;                  // 200000
  maxOutput: number;                      // 8192
  supportedFeatures: ('completion' | 'streaming' | 'embedding')[];
  costPer1kInput: number;                 // USD
  costPer1kOutput: number;                // USD
}
```

### 지원 Provider

| Provider | 모델 | 연결 방식 | 특징 |
|----------|------|-----------|------|
| **Claude** (Anthropic) | Sonnet 4.5, Haiku 4.5, Opus 4.5 | HTTPS API | 기본 추천 Provider |
| **OpenAI** | GPT-4o, GPT-4o-mini, o1 | HTTPS API | 광범위한 사용자 기반 |
| **Google Gemini** | Gemini 2.0 Flash, Pro | HTTPS API | 무료 티어 가용 |
| **Local LLM** | Ollama / llama.cpp 모델 | 로컬 HTTP | 오프라인, Privacy |

**Rust 백엔드 프록시**

LLM API 호출은 프론트엔드에서 직접 수행하지 않고, Part 3(§3.2)의 Rust 백엔드를 통해 프록시한다.

```
[LLM 호출 흐름]

  프론트엔드 (React)
       │
       │ Tauri IPC: invoke('llm_complete', { prompt, model, config })
       │
       ▼
  Rust 백엔드 (Tauri Command)
       │
       ├── API 키 복호화 (OS Keychain에서 읽기)
       ├── 요청 구성 (헤더, 모델 파라미터)
       ├── Provider별 HTTP 클라이언트 호출
       │
       │ Tauri Event: emit('llm-chunk', { request_id, chunk, done })
       │
       ▼
  프론트엔드 (이벤트 수신)
       │
       └── Ghost Text / Inline Diff / Chat Panel 갱신
```

이 구조의 이점은 세 가지이다. 첫째, API 키가 프론트엔드 메모리에 노출되지 않는다. 둘째, CORS 제약 없이 모든 Provider에 접근한다. 셋째, 로컬 LLM(Ollama)과 원격 API를 동일한 인터페이스로 처리한다.

### 모델 선택

**수동 선택**: 설정(Part 4, §4.8)의 AI 탭에서 Provider + 모델을 직접 선택한다.

**Auto 모드**: 작업 유형에 따라 자동으로 적절한 모델을 선택한다.

| 작업 | 권장 모델 선택 기준 | 사유 |
|------|---------------------|------|
| Ghost Text | 가장 빠른 모델 (Haiku, Flash, GPT-4o-mini) | 레이턴시 최우선 |
| 인라인 편집 | 중간 모델 (Sonnet, GPT-4o) | 품질/속도 균형 |
| Chat | 설정된 기본 모델 | 사용자 선호 |
| Agent Mode | 가장 강력한 모델 (Opus, o1) | 복잡한 추론 필요 |
| Knowledge Q&A | 중간 모델 + 임베딩 모델 | RAG 파이프라인 |

Auto 모드의 매핑은 사용자가 설정에서 커스터마이징할 수 있다.

### Privacy Mode

민감한 문서를 다루는 사용자를 위한 제로 데이터 보존 옵션이다.

```
[Privacy Mode 활성 시]

  ✅ 허용:
  ├── 로컬 LLM (Ollama) 사용
  ├── API 호출 시 "no-store" 헤더 전송 (지원하는 Provider)
  └── 에디터 기본 기능 전체

  ❌ 차단:
  ├── Knowledge Q&A 임베딩 외부 전송
  ├── 대화 히스토리의 클라우드 동기화
  └── 문서 내용이 포함된 텔레메트리
```

Privacy Mode는 전역 설정 또는 특정 폴더/파일 단위로 적용할 수 있다. frontmatter에 `privacy: true`를 설정하면 해당 파일에 대해서만 Privacy Mode가 적용된다.

### API 키 관리

Part 3(§3.5)에서 정의한 대로, API 키는 Tauri의 OS Keychain 통합으로 암호화 저장한다.

```
[API 키 저장 흐름]

  사용자가 설정 UI에서 API 키 입력
       │
       ▼
  프론트엔드 → IPC → Rust 백엔드
       │
       ▼
  OS Keychain에 암호화 저장
  ├── macOS: Keychain Services
  ├── Windows: Credential Manager
  └── Linux: libsecret (GNOME Keyring / KDE Wallet)
       │
       ▼
  settingsStore.ai.apiKey = "***" (마스킹된 표시용 값만 저장)
```

설정 JSON 파일에는 실제 API 키가 저장되지 않는다. `"apiKey": "keychain:baram-claude-api-key"` 형태의 참조만 저장한다.

## 6.4 Skills-Aware AI 기능 (Baram 차별화)

Part 1(§1.3)에서 정의한 Persona 1(LLM/AI 개발자)을 위한 Baram 고유의 차별화 기능이다. Part 5(§5.8)의 LLM Skills 편집 모드와 연동하여 AI가 스킬 파일의 구조와 의미를 이해한다.

### 스킬 구조 자동 생성

목적만 설명하면 AI가 frontmatter + 섹션 구조 + 프롬프트 초안을 생성한다.

```
[스킬 자동 생성 흐름]

  커맨드 팔레트: "Skills: Create New Skill"
       │
       ▼
  목적 입력 대화상자
  ┌──────────────────────────────────────────┐
  │ 어떤 스킬을 만들까요?                       │
  │                                          │
  │ [PDF를 읽고 핵심 내용을 요약하는 스킬       ]│
  │                                          │
  │ 📁 저장 위치: skills/summarizer.md         │
  │ 📋 템플릿: prompt (자동 선택)               │
  │                                    [생성] │
  └──────────────────────────────────────────┘
       │
       ▼
  AI 생성 결과 (에디터에 삽입, diff 미리보기)

  ---
  name: pdf-summarizer
  description: PDF 문서를 읽고 핵심 내용을 구조화된 요약으로 생성
  version: 1.0.0
  tags: [summarization, pdf, extraction]
  requires: [base-skill]
  input_format: pdf
  output_format: markdown
  ---

  # PDF Summarizer

  <system>
  You are a document summarization specialist...
  </system>

  <instructions>
  Given the following PDF content, create a structured summary...
  </instructions>

  <output_format>
  ## Summary
  - Key findings: ...
  - Main arguments: ...
  - Conclusions: ...
  </output_format>
```

### 프롬프트 최적화 제안

스킬 파일 편집 중 AI가 프롬프트 품질을 분석하고 개선점을 제안한다. Part 5(§5.8)의 프롬프트 구문 하이라이팅과 연동된다.

```
[프롬프트 최적화 — 인라인 경고]

  <instructions>
  요약해주세요.                              ← ⚠️ 밑줄
  </instructions>                               │
                                                 ▼
                              ┌──────────────────────────────┐
                              │ ⚠️ 모호한 지시                 │
                              │                              │
                              │ "요약해주세요"는 출력 형식,     │
                              │ 길이, 초점이 불명확합니다.      │
                              │                              │
                              │ 💡 제안:                      │
                              │ "다음 내용을 3개의 핵심 포인트로│
                              │  요약하세요. 각 포인트는 1-2    │
                              │  문장으로 작성하세요."          │
                              │                              │
                              │ [적용] [무시] [모두 검토]       │
                              └──────────────────────────────┘
```

**검출하는 문제 유형**:

| 경고 유형 | 감지 조건 | 제안 내용 |
|-----------|-----------|-----------|
| 모호한 지시 | 구체적 동사/조건 없는 짧은 지시 | 구체적 지시로 개선 |
| 출력 형식 누락 | `<output_format>` 미정의 | 형식 섹션 추가 |
| 필수 필드 누락 | name/description 없음 | 필드 추가 |
| 과도한 길이 | 프롬프트 토큰이 컨텍스트 윈도우의 50% 초과 | 분리 또는 축약 |
| 변수 미사용 | `{{variable}}` 정의했지만 본문에서 미참조 | 사용 또는 제거 |
| 충돌 지시 | 상충되는 두 개 이상의 지시가 존재 | 일관성 검토 |

커맨드 팔레트에서 "Skills: Review Prompt Quality" 실행 시, 현재 파일의 모든 문제를 한 번에 분석하여 목록으로 표시한다.

### 스킬 인라인 테스트 실행

에디터 안에서 스킬을 즉시 LLM에 호출하고 결과를 미리볼 수 있다.

```
[인라인 테스트 실행 — 커맨드 팔레트 또는 단축키]

  "Skills: Test Current Skill" (Cmd+Shift+T)
       │
       ▼
  테스트 입력 대화상자
  ┌──────────────────────────────────────────┐
  │ 테스트 입력                                │
  │                                          │
  │ {{input}} 변수에 전달할 내용:              │
  │ ┌──────────────────────────────────────┐  │
  │ │ Baram은 Tauri 기반의 마크다운 에디터로,│  │
  │ │ WYSIWYG 편집과 AI 통합을 특징으로...  │  │
  │ └──────────────────────────────────────┘  │
  │                                          │
  │ 모델: [Claude Sonnet 4.5 ▾]              │
  │                                          │
  │              [실행] [취소]                  │
  └──────────────────────────────────────────┘
       │
       ▼
  결과 — 에디터 하단 패널에 표시
  ┌──────────────────────────────────────────┐
  │ 📊 테스트 결과                              │
  │ 모델: Claude Sonnet 4.5 | 토큰: 1,247     │
  │ 시간: 2.3s | 비용: $0.004                  │
  │ ──────────────────────────────────────── │
  │                                          │
  │ ## Summary                               │
  │ - Key findings: Baram은 Tauri 2.0 기반... │
  │ - Main arguments: WYSIWYG와 AI의 통합...  │
  │ - Conclusions: ...                        │
  │                                          │
  │ [📋 복사] [📝 새 파일로 저장] [↻ 재실행]     │
  └──────────────────────────────────────────┘
```

테스트 결과 패널은 에디터 하단에 분할 표시되며, 스킬 소스와 결과를 동시에 볼 수 있다. 여러 번 실행 시 이전 결과와 비교할 수 있는 히스토리 탭이 제공된다.

### 스킬 간 의존성 분석 AI

`requires` 필드로 연결된 스킬들의 인터페이스 호환성을 AI가 분석한다.

```
[의존성 분석 — 자동 감지]

  skills/analyzer.md
  ---
  requires: [base-skill, pdf-reader]
  ---

  AI가 감지:
  ┌────────────────────────────────────────────┐
  │ ⚠️ 의존성 경고                               │
  │                                            │
  │ pdf-reader.md의 output_format이 "json"이지만│
  │ analyzer.md는 "markdown" 입력을 기대합니다.  │
  │                                            │
  │ 💡 pdf-reader의 출력을 markdown으로 변환하는  │
  │    중간 스킬을 추가하거나, analyzer의 입력     │
  │    형식을 json으로 변경하세요.                │
  │                                            │
  │ [자동 수정] [무시]                            │
  └────────────────────────────────────────────┘
```

### LLM 해석 시뮬레이션

동일한 스킬 프롬프트를 여러 LLM 모델로 실행하여 결과를 나란히 비교한다.

```
[LLM 해석 시뮬레이션 — side-by-side 비교]

  "Skills: Compare Models" 실행
       │
       ▼
  모델 선택 (복수 선택 가능)
  ┌────────────────────────────────┐
  │ ☑ Claude Sonnet 4.5            │
  │ ☑ GPT-4o                       │
  │ ☐ Gemini 2.0 Flash             │
  │ ☑ Ollama (llama3)              │
  │                          [실행] │
  └────────────────────────────────┘
       │
       ▼
  결과 비교 뷰 (수평 분할)
  ┌─────────────────┬──────────────────┬──────────────────┐
  │ Claude Sonnet    │ GPT-4o           │ Ollama llama3    │
  │ 토큰: 1,247      │ 토큰: 1,089      │ 토큰: 1,432      │
  │ 시간: 2.3s       │ 시간: 1.8s       │ 시간: 4.1s       │
  │ 비용: $0.004     │ 비용: $0.003     │ 비용: $0.000     │
  │ ───────────────│──────────────── │──────────────── │
  │                 │                  │                  │
  │ ## Summary      │ ## Summary       │ ## Summary       │
  │ ...             │ ...              │ ...              │
  │                 │                  │                  │
  └─────────────────┴──────────────────┴──────────────────┘
```

이 기능은 스킬 개발자가 프롬프트의 모델 간 일관성을 검증하고, 최적의 모델을 선택하는 데 활용한다.
