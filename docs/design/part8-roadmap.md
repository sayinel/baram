## 8.1 Phase 정의 (3단계)

### 전체 구조 요약

```
Phase 1: MVP          Phase 2: 확장 기능          Phase 3: 고급 기능
(핵심 에디터)          (연결·AI·생산성)            (지식관리·협업·생태계)
 ~6개월                ~6개월                     ~6개월

 27개 기능              30개 기능                  15개 기능
──────────────────────────────────────────────────────────────
 M1─M2─M3─M4─M5─M6    M7────M8────M9             M10
```

총 72개 기능을 3개 Phase, 10개 마일스톤으로 나누어 18개월에 걸쳐 구현한다. 각 Phase는 독립적으로 릴리스 가능한 완결된 제품 상태를 목표로 한다.

### Phase 1: MVP (핵심 에디터)

Phase 1의 목표는 "Typora 대체재"이다. 파일을 열고, 마크다운을 WYSIWYG로 편집하고, 저장하고, 내보내는 기본 루프가 완벽하게 동작해야 한다. AI Level 2(인라인 편집)를 포함하여 경쟁 제품 대비 차별점을 초기부터 확보한다.

```
에디터 코어 (8개):
  01. WYSIWYG 에디터 코어 (Tiptap + ProseMirror)          ← M2
  02. 마크다운 파싱 ↔ 직렬화 파이프라인                      ← M2
  03. 기본 블록 요소 (H1~H6, 인용, 리스트, 체크리스트, 구분선) ← M2
  04. 기본 인라인 요소 (볼드, 이탤릭, 코드, 링크, 이미지)      ← M2
  05. KaTeX 수식 렌더링 (인라인 + 블록)                      ← M3
  06. 코드 블록 (CodeMirror 6 구문 강조)                     ← M3
  07. 테이블 에디터                                          ← M3
  08. Source Code Mode 전환                                  ← M3

UI/UX (9개):
  09. 3-Layer Interaction 기반 레이아웃                       ← M4
  10. 커맨드 팔레트 (Cmd+K)                                  ← M4
  11. 슬래시 커맨드 (/)                                      ← M4
  12. 플로팅 서식 툴바 (텍스트 선택 시)                        ← M4
  13. 블록 핸들 메뉴 (좌측 호버)                               ← M4
  14. 컨텍스트 메뉴 (7종)                                     ← M4
  15. 파일 트리 사이드바                                       ← M4
  16. 아웃라인 사이드바 (필터 포함)                             ← M4
  17. 상태바 (미니멀)                                         ← M4

파일 (3개):
  18. 파일 열기/저장/생성                                      ← M2
  19. 자동 저장 (Debounced Write)                             ← M2
  20. YAML Frontmatter 지원                                   ← M3

AI Level 2 (5개):
  21. LLM Provider 추상화 레이어 + Rust 프록시                  ← M5
  22. AI 설정 UI                                               ← M5
  23. 슬래시 AI 커맨드 7종                                      ← M5
  24. Cmd+K 인라인 편집 + AI Diff Engine                       ← M5
  25. 플로팅 툴바 AI 버튼 (✨)                                  ← M5

내보내기 (2개):
  26. PDF 내보내기                                              ← M6
  27. HTML 내보내기                                             ← M6
```

**Phase 1 완료 기준**: 사용자가 Baram을 설치하고, 마크다운 파일을 WYSIWYG로 편집하며, AI 인라인 편집을 사용하고, PDF/HTML로 내보낼 수 있다. Typora의 핵심 기능을 모두 대체하면서, AI 편집이라는 추가 가치를 제공한다.

### Phase 2: 확장 기능

```

  29. 백링크 패널                                               ← M7
  30. 블록 참조 ((())) + 블록 임베드 ({{embed}})                ← M7
  31. 링크 자동완성 (별칭 매칭 포함)                             ← M7
  32. 호버 미리보기                                             ← M7
  33. 파일 이름 변경 시 링크 자동 갱신                            ← M7
  34. 언링크드 멘션                                             ← M7

네비게이션 (5개):

  39. 문서 전환 (Ctrl+Tab)                                      ← M7

AI Level 1 + 3 + Skills-Aware (7개):
  43. Ghost Text 제안 (디바운스, 컨텍스트 모드, 3-tier 프롬프트) ← M8
  44. AI Chat Panel (사이드바, @참조, Apply to Editor)           ← M8
  45. Skills 구조 자동 생성                                     ← M8
  46. 프롬프트 최적화 제안                                       ← M8
  47. 스킬 인라인 테스트 실행                                    ← M8

기타 (8개):
  50. Mermaid 다이어그램                                        ← M9

  54. 테마 시스템 (커스텀 CSS)                                    ← M9
  55. Pandoc 연동 확장 내보내기 (Word, LaTeX, Epub)               ← M9
  56. 저널/데일리 노트                                           ← M9

Git 기본 (Phase 2에 추가, §5.15):
```

**Phase 2 완료 기준**: 양방향 링크로 지식을 연결하고, AI Chat으로 문서를 대화하며, Skills 파일을 전문적으로 편집하고 테스트할 수 있다. Obsidian 사용자가 전환할 수 있을 만큼의 연결 기능이 갖춰져 있다.

### Phase 3: 고급 기능

Phase 3의 목표는 "독자적 생태계 구축"이다. Agent Mode, 실시간 협업, 플러그인 마켓플레이스로 장기적 경쟁력을 확보한다.

```
지식 관리 (4개):
  58. 그래프 뷰                                                 ← M10
  59. 쿼리 블록 (비주얼 쿼리 빌더)                                ← M10
  60. Canvas (무한 캔버스)                                       ← M10
  61. 네임스페이스                                                ← M10

AI Level 4 + 5 (5개):
  62. Agent Mode (멀티파일 자율 편집, 계획→실행→diff)              ← M10
  63. Knowledge Q&A (벡터 검색, citation)                        ← M10
  64. LLM 해석 시뮬레이션 (모델별 비교)                            ← M10
  65. MCP 서버 연동                                               ← M10
  66. 커스텀 AI 플러그인 API                                      ← M10

협업 & 인프라 (5개):
  67. Git 고급 (원격 동기화, PR/MR, 충돌 해결 UI)                  ← M10
  68. 실시간 협업 (Yjs CRDT)                                      ← M10
  69. 플러그인 마켓플레이스                                        ← M10
  70. 모바일 대응 (Tauri Mobile)                                  ← M10
  71. 파일 스냅샷 / 버전 히스토리 UI                                ← M10

LLM Skills 전용 (1개):
  72. LLM Skills 전용 모드 (전체 UI 최적화)                        ← M10
```

**Phase 3 완료 기준**: AI Agent가 여러 파일을 자율적으로 편집하고, 팀이 실시간 협업하며, 커뮤니티 플러그인으로 기능이 확장되는 성숙한 제품 상태.

### M1: 프로젝트 셋업 (Week 1\~2)

**목표**: 개발 환경과 프로젝트 스캐폴딩을 완료하고, 빈 에디터 창이 실행되는 상태.

| 산출물                | 상세                                       |
| ------------------ | ---------------------------------------- |
| Tauri 2.0 프로젝트 초기화 | Rust 백엔드 + React 프론트엔드 스캐폴딩              |
| 빌드 파이프라인           | Vite 6 + TypeScript + Tailwind CSS 설정    |
| Tiptap 에디터 임베드     | 기본 에디터 인스턴스가 빈 창에 렌더링                    |
| Rust 백엔드 뼈대        | IPC 커맨드 구조, 파일 I/O 기본 모듈                 |
| CI/CD              | GitHub Actions — 빌드, 린트, 테스트, 크로스 플랫폼 빌드 |
| 프로젝트 구조            | 디렉토리 구조, 코딩 컨벤션, 기본 컴포넌트 디자인 시스템         |

**완료 기준**: macOS/Windows/Linux에서 빈 에디터 창이 실행되고, 텍스트 입력이 가능하다.

### M2: 기본 편집 (Week 3\~8)

**목표**: 마크다운 파일을 열고, WYSIWYG로 편집하고, 저장할 수 있다.

| 산출물                    | 상세                                                                                              |
| ---------------------- | ----------------------------------------------------------------------------------------------- |
| 마크다운 파이프라인             | remark-parse → mdast → ProseMirror 양방향 변환                                                       |
| Core Editing Extension | Heading, Paragraph, Bold, Italic, Code, Link, Image, Blockquote, List, TaskList, HorizontalRule |
| InputRule 엔진           | `#`→헤딩, `**`→볼드, `>`→인용 등 마크다운 구문 자동 변환                                                         |
| PasteRule 엔진           | HTML/마크다운 붙여넣기 감지 및 변환                                                                          |
| 파일 열기/저장               | Tauri fs API, 원자적 쓰기, 기본 파일 다이얼로그                                                               |
| 자동 저장                  | 디바운스 2초, dirty 상태 표시                                                                            |
| History                | Undo/Redo (ProseMirror 내장)                                                                      |

**완료 기준**: `.md` 파일을 열면 WYSIWYG로 표시되고, 편집 후 저장하면 정확한 마크다운이 출력된다. 마크다운 구문 입력 시 즉시 렌더링으로 변환된다.

### M3: 리치 콘텐츠 (Week 9\~14)

**목표**: KaTeX 수식, 코드 블록, 테이블, Frontmatter 등 복잡한 콘텐츠 유형이 동작한다.

| 산출물                | 상세                                                           |
| ------------------ | ------------------------------------------------------------ |
| KaTeX 수식           | 인라인(`$...$`) + 블록(`$$...$$`), 라이브 프리뷰 오버레이(§5.3), LaTeX 자동완성 |
| CodeMirror 6 코드 블록 | NodeView 임베드, 구문 강조 20개 언어, 언어 선택 드롭다운                       |
| 테이블 에디터            | 셀 편집, 행/열 추가·삭제, 정렬, Tab 네비게이션, 플로팅 컨트롤바                     |
| Frontmatter        | YAML 파싱/직렬화, 소스 뷰, YAML 순서/주석 보존                             |
| Source Code Mode   | Cmd+/ 토글, 마크다운 소스 ↔ WYSIWYG 전환                               |

**완료 기준**: KaTeX 수식이 렌더링되고 편집 가능하며, 코드 블록에서 구문 강조가 동작하고, 테이블을 직관적으로 조작할 수 있다.

### M4: UI 프레임워크 (Week 15\~18)

**목표**: 3-Layer Interaction 기반 전체 UI가 완성된다.

| 산출물          | 상세                                           |
| ------------ | -------------------------------------------- |
| 3-Layer 레이아웃 | 에디터 중심, 좌 사이드바, 우 패널 리사이즈 가능                 |
| 커맨드 팔레트      | Cmd+K 트리거, 퍼지 검색, MRU 정렬, 카테고리 필터            |
| 슬래시 커맨드      | `/` 입력 시 블록 삽입 메뉴 (14종 기본 + 2종 연결 + 1종 AI)   |
| 플로팅 서식 툴바    | 텍스트 선택 시 인라인 서식 12종                          |
| 블록 핸들 메뉴     | 좌측 호버 ⋮ 아이콘, 드래그, 블록 변환 메뉴                   |
| 컨텍스트 메뉴      | 7종 (에디터, 파일트리, 탭, 이미지, 코드, 테이블, 링크)          |
| 파일 트리 사이드바   | 폴더 트리, 파일 생성/삭제/이름변경, 드래그 이동                 |
| 아웃라인 사이드바    | 헤딩 목차, 레벨 필터, 클릭 스크롤                         |
| 상태바          | Git 브랜치(비활성), 워드카운트, 행:열                     |
| 메뉴바          | 6-메뉴 구조 (File, Edit, Insert, Go, View, Help) |
| 단축키 시스템      | Part 4(§4.6) 전체 단축키 맵, keybindings.json 로드   |
| 다크/라이트 테마    | system/light/dark, CSS Variables 기반          |

**완료 기준**: 커맨드 팔레트, 슬래시 커맨드, 플로팅 툴바가 모두 동작한다. 사이드바 파일 트리에서 파일을 선택하면 에디터에 열린다. 다크/라이트 테마가 전환된다.

### M5: AI Level 2 (Week 19\~22)

**목표**: AI 인라인 편집이 동작한다. API 키를 설정하고, 텍스트를 선택하여 AI로 개선/번역/요약할 수 있다.

| 산출물              | 상세                                                                                                     |
| ---------------- | ------------------------------------------------------------------------------------------------------ |
| LLM Provider 추상화 | TypeScript 인터페이스 + 4개 프로바이더 어댑터 (Claude, OpenAI, Gemini, Ollama)                                       |
| Rust 프록시         | IPC → API 키 복호화 → HTTP 요청 → 스트리밍 Event                                                                 |
| API 키 관리         | OS Keychain 저장, 설정 UI에서 입력/테스트                                                                         |
| AI 설정 UI         | 프로바이더 선택, 모델 드롭다운, 연결 테스트                                                                              |
| 플로팅 툴바 AI 버튼     | ✨ 버튼 → 6개 프리셋 (개선, 축소, 확장, 번역, 톤, 설명)                                                                  |
| Cmd+K 인라인 편집     | 자연어 지시 → 스트리밍 → diff 미리보기 → 수락/거부                                                                      |
| AI Diff Engine   | 문자 수준 diff, ProseMirror Decoration (초록 삽입, 빨간 삭제), 청크별 부분 수락                                           |
| 슬래시 AI 커맨드       | 7종 (/ai-write, /ai-brainstorm, /ai-translate, /ai-summarize, /ai-expand, /ai-fix-grammar, /ai-explain) |

**완료 기준**: 텍스트를 선택하고 "번역: 영어"를 입력하면, 스트리밍으로 번역 결과가 diff로 표시되고, 수락/거부할 수 있다.

### M6: MVP 릴리스 (Week 23\~26)

**목표**: Phase 1 전체를 통합하여 퍼블릭 릴리스 가능 상태를 만든다.

| 산출물       | 상세                                                    |
| --------- | ----------------------------------------------------- |
| PDF 내보내기  | Tauri webview → PDF, KaTeX/코드/테이블 포함                  |
| HTML 내보내기 | 스탠드얼론 HTML, CSS 인라인, 이미지 임베드 옵션                       |
| 통합 테스트    | E2E 테스트 (Playwright), 크로스 플랫폼 검증                      |

## 8.3 기술 리스크 및 대응 계획

### 리스크 매트릭스

```
[영향도 × 발생 가능성]

  영향도
  높음  │  ③ 직렬화 정확도   │  ① WebView 호환성    │
        │  ⑤ Ghost Text 품질  │  ④ LLM 비용/레이턴시  │
  ───── ├────────────────── ├─────────────────── │
  중간  │  ⑦ 임베딩 저장 용량  │  ② 대용량 파일 성능   │
        │                   │  ⑥ 크로스 플랫폼 테스트 │
  ───── ├────────────────── ├─────────────────── │
  낮음  │                   │  ⑧ Tauri 2.0 안정성   │
        └────────────────── └─────────────────── 
          낮음                  높음
                    발생 가능성
```

### 상세 리스크 및 대응

**① Tauri WebView 크로스 플랫폼 호환성** — 발생 가능성: 높음, 영향도: 높음

macOS(WebKit), Windows(WebView2), Linux(WebKitGTK)의 렌더링 차이로 CSS/JavaScript 동작이 플랫폼마다 다를 수 있다. 특히 WebKitGTK는 macOS WebKit보다 버전이 뒤처지는 경우가 있다.

| 대응            | 상세                                                 |
| ------------- | -------------------------------------------------- |
| 조기 검증         | M1부터 3개 플랫폼 동시 빌드/테스트 자동화                          |
| 기능 탐지         | CSS `@supports`, JS feature detection으로 폴백 구현      |
| 폴리필           | 누락된 Web API는 polyfill 또는 Tauri 플러그인으로 보완           |
| 테스트 매트릭스      | macOS 14+, Windows 10/11, Ubuntu 22.04/24.04 최소 커버 |
| 최소 WebView 버전 | WebView2 Runtime 120+, WebKitGTK 2.42+ 명시          |

**② 대용량 파일 성능** — 발생 가능성: 높음, 영향도: 중간

10,000줄 이상 마크다운 파일에서 ProseMirror의 DOM 업데이트 성능이 저하될 수 있다. 수백 개의 수식/코드 블록이 있는 기술 문서에서 특히 문제가 된다.

| 대응       | 상세                                           |
| -------- | -------------------------------------------- |
| 가상 스크롤   | 화면 밖 블록의 NodeView를 placeholder로 대체           |
| 지연 렌더링   | KaTeX, Mermaid는 viewport에 진입할 때만 렌더링         |
| 트랜잭션 최적화 | 대량 변경 시 batch transaction 사용                 |
| 성능 벤치마크  | M3에서 5,000줄/10,000줄/50,000줄 파일 기준 FPS 측정     |
| 제한 가이드   | 성능 저하 시 "이 파일은 Source Code Mode에서 더 빠릅니다" 안내 |

**③ 마크다운 직렬화 정확도** — 발생 가능성: 높음, 영향도: 높음

ProseMirror 문서 → 마크다운 변환 시 원본 서식이 손실되거나 변형될 수 있다. 중첩 리스트, 테이블 안의 인라인 서식, 연속 빈 줄 등이 특히 어렵다.

| 대응            | 상세                                        |
| ------------- | ----------------------------------------- |
| 라운드트립 테스트     | MD → Editor → MD가 동일한 결과를 내는지 자동 테스트      |
| 퍼징 테스트        | 임의의 마크다운 입력으로 파서/직렬화기 경계 조건 탐색            |
| GFM 스펙 테스트 슈트 | CommonMark 공식 테스트 651개 + GFM 확장 테스트 전수 통과 |
| 에지 케이스 수집     | 실사용 마크다운 파일 1,000개+ 수집하여 회귀 테스트           |
| 직렬화 규칙        | §7.1의 정규화 규칙으로 출력 일관성 보장                  |

**④ LLM API 레이턴시 및 비용** — 발생 가능성: 높음, 영향도: 높음

Ghost Text의 500ms 디바운스 후 API 호출에 추가로 500ms\~2초의 네트워크 지연이 발생하면 UX가 저하된다. 활발한 사용 시 월 \$50+ 비용이 발생할 수 있다.

| 대응        | 상세                                               |
| --------- | ------------------------------------------------ |
| 최소 모델 사용  | Ghost Text는 가장 빠른 모델 (Haiku, GPT-4o-mini, Flash) |
| 프리페치      | 커서 위치 기반 다음 제안 미리 요청                             |
| 접두사 캐시    | 동일 접두사의 이전 응답 재사용                                |
| 스트리밍      | 첫 토큰 즉시 표시, 전체 완료 불필요                            |
| 토큰 제한     | Ghost Text 기본 max\_tokens: 100, 인라인 편집: 500      |
| 비용 대시보드   | 설정 → AI 탭에서 월간 사용량/추정 비용 표시                      |
| 로컬 LLM 대안 | Ollama/llama.cpp로 비용 제로 옵션 제공                    |
| 자동 비활성화   | 월간 토큰 한도 설정, 초과 시 Ghost Text 자동 중지               |

**⑤ Ghost Text 제안 품질** — 발생 가능성: 높음, 영향도: 높음

불필요하거나 부정확한 Ghost Text가 자주 나타나면 사용자가 기능을 비활성화하고 부정적 인상을 받는다.

| 대응       | 상세                                               |
| -------- | ------------------------------------------------ |
| 컨텍스트 최적화 | 3-tier 프롬프트 (현재 파일 → 열린 파일 → Vault), 컨텍스트별 특화 모드 |
| 빈도 조절    | 연속 거부 시 자동으로 빈도 축소 (학습 기반)                       |
| 쉬운 비활성화  | 상태바 1-click 토글, Cmd+Shift+G 단축키                  |
| A/B 테스트  | 프롬프트 변형별 수락률 측정 (텔레메트리 옵트인)                      |
| 사용자 피드백  | Ghost Text 거부 시 선택적 피드백 수집                       |

**⑥ 크로스 플랫폼 테스트 전략** — 발생 가능성: 높음, 영향도: 중간

3개 OS × 다수 버전 조합에서 수동 테스트는 비현실적이다.

| 대응      | 상세                                                  |
| ------- | --------------------------------------------------- |
| CI 매트릭스 | GitHub Actions — macOS 14, Windows 11, Ubuntu 24.04 |
| E2E 자동화 | Playwright + Tauri WebDriver로 주요 시나리오 자동 테스트        |
| 스크린샷 비교 | 플랫폼별 렌더링 스냅샷 비교 (시각적 회귀 감지)                         |
| Beta 채널 | 얼리 어답터 그룹을 통한 실사용 피드백                               |

**⑦ 벡터 임베딩 저장 용량** — 발생 가능성: 중간, 영향도: 중간

대형 Vault(10,000+ 파일)에서 임베딩 인덱스가 수백 MB로 커질 수 있다.

| 대응     | 상세                                    |
| ------ | ------------------------------------- |
| 증분 갱신  | 변경 파일만 재임베딩, 전체 재구축 최소화               |
| 용량 제한  | `ai.embeddingMaxStorageMB` (기본 200MB) |
| 자동 정리  | 30일 이상 미사용 청크 자동 삭제                   |
| 로컬 임베딩 | Ollama 임베딩 모델로 API 비용 없이 생성 가능        |

**⑧ Tauri 2.0 프레임워크 안정성** — 발생 가능성: 낮음, 영향도: 중간

Tauri 2.0은 비교적 새로운 릴리스이며, 특정 플러그인이나 API에서 예기치 않은 버그가 있을 수 있다.

| 대응          | 상세                                     |
| ----------- | -------------------------------------- |
| Tauri 버전 고정 | 안정 릴리스에 고정, 패치 업데이트만 추적                |
| 대안 준비       | 핵심 기능에 Tauri 전용 API 의존 최소화, 추상화 레이어 유지 |
| 커뮤니티 모니터링   | Tauri Discord/GitHub Issues 적극 추적      |

## 8.4 품질 보증 전략

### 테스트 피라미드

```
              ▲
             / \
            / E2E \           ← Playwright + Tauri WebDriver
           / (10%) \              macOS/Windows/Linux 매트릭스
          /─────────\
         / Integration \      ← Rust + TS 통합 테스트

      /     Unit Tests     \  ← Jest (TS) + cargo test (Rust)
     /      (70%)           \     파서, 직렬화, 스키마, 상태
    /─────────────────────────\
```

### 핵심 테스트 영역

| 영역              | 테스트 유형 | 도구          | 기준                                  |
| --------------- | ------ | ----------- | ----------------------------------- |
| 마크다운 라운드트립      | 단위     | Jest        | CommonMark 651개 + GFM + 확장 구문 전수 통과 |

| Rust 파일 I/O     | 단위     | cargo test  | 원자적 쓰기, File Watcher, 링크 인덱싱        |
| Rust 검색 엔진      | 단위     | cargo test  | tantivy 인덱싱/쿼리, 한글 토큰화              |
| IPC 커맨드         | 통합     | Tauri test  | 모든 IPC 커맨드 요청/응답                    |
| LLM 프록시         | 통합     | mock server | 스트리밍, 에러 핸들링, 타임아웃                  |
| UI 인터랙션         | E2E    | Playwright  | 파일 열기→편집→저장, AI 인라인 편집, 탭 전환        |

### 성능 벤치마크

## 8.5 릴리스 전략

### 버전 체계

Semantic Versioning (SemVer)을 따른다.

```
v{MAJOR}.{MINOR}.{PATCH}[-{PRE-RELEASE}]

v0.1.0     ← Phase 1 MVP 첫 릴리스 (공개 Beta)
v0.2.0     ← Phase 1 기능 추가
v0.5.0     ← Phase 2 첫 릴리스
v0.9.0     ← Phase 2 완성
v1.0.0     ← Phase 3 시작 시점 (안정 릴리스)
v1.x.x     ← Phase 3 점진적 기능 추가
```

MAJOR 1.0 미만(0.x.x)은 API와 데이터 형식이 변경될 수 있다. 1.0.0부터 하위 호환성을 보장한다.

### 릴리스 채널

| 채널          | 주기      | 대상       | 안정성          |
| ----------- | ------- | -------- | ------------ |
| **Nightly** | 매일 (자동) | 개발자, 기여자 | 불안정          |
| **Beta**    | 2주마다    | 얼리 어답터   | 기능 완성, 버그 가능 |
| **Stable**  | 4\~8주마다 | 일반 사용자   | 검증 완료        |

### 자동 업데이트

Tauri의 내장 업데이트 기능(`tauri-plugin-updater`)을 사용한다. Stable 채널은 자동 업데이트(확인 후 설치), Beta/Nightly는 수동 확인 방식이다.

```
업데이트 흐름:
  앱 시작 → 업데이트 서버 확인 → 새 버전 있음 →
  알림 표시 "v0.3.0 available" →
  [지금 업데이트] [나중에] [이 버전 건너뛰기]
```

## 8.6 Phase별 핵심 의존성 체인

각 Phase 내에서 기능 간 의존 관계와 병렬 작업 가능 여부를 나타낸다.

### Phase 1 의존성

```
[M1] 프로젝트 셋업
  │
  ▼
[M2] 기본 편집 ─────────────────────────┐
  │ (Tiptap, 파이프라인, InputRule)       │
  │                                     │
  ├──→ [M3] 리치 콘텐츠 (수식, 코드, 테이블)│  ← M2 완료 후 시작
  │                                     │
  └──→ [M4] UI 프레임워크 ──────────────┘  ← M2와 부분 병렬 가능
         │                                  (레이아웃/사이드바는 독립)
         ▼
       [M5] AI Level 2 ─────────────────── ← M4의 플로팅 툴바 필요
         │
         ▼
       [M6] MVP 릴리스 ─────────────────── ← M2~M5 모두 완료 필요
```

**병렬화 기회**: M3(리치 콘텐츠)와 M4(UI 프레임워크)는 대부분 병렬 작업 가능하다. M3의 NodeView 구현과 M4의 레이아웃/사이드바는 독립적이다.

### Phase 2 의존성

```
[M7] 연결 + 네비게이션 ─────────────┐
  │ (링크 인덱스가 핵심 의존성)       │
  │                                │
  └──→ [M8] AI 심화 + Skills ──────┘  ← M7의 인덱스 활용 (@참조)
         │                              Ghost Text는 M7과 병렬 가능
         ▼
       [M9] 생산성 도구 ──────────────── ← M7 인덱스 + M8 AI 기반
                                         Mermaid, 테마는 독립 작업 가능
```

**병렬화 기회**: M8의 Ghost Text는 M7과 병렬 개발 가능 (인덱스 불필요). M9의 Mermaid, 테마 시스템, Notion 가져오기는 독립적이므로 M7/M8과 병렬 가능하다.

### Phase 3 의존성

Phase 3는 기능이 대부분 독립적이므로 우선순위 기반으로 순서를 결정한다. 사용자 피드백에 따라 순서를 조정할 수 있다.

```
[독립 작업 가능]
  · 그래프 뷰 (링크 인덱스만 필요 — M7에서 완료)
  · 쿼리 블록 (인덱스 + tantivy — M7/M9에서 완료)
  · Git 고급 (Git 기본 — M9에서 완료)
  · 스냅샷 UI (Rust 백엔드 — 독립)

[의존성 있음]
  · Agent Mode → AI Level 2 + Chat 완료 필요
  · Knowledge Q&A → 임베딩 인덱스 구현 필요
  · 플러그인 마켓플레이스 → Extension API 안정화 필요
  · 실시간 협업 → Yjs CRDT 통합, 서버 인프라
  · Canvas → ProseMirror 외 별도 렌더링 엔진
```
