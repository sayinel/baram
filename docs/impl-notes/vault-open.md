# Vault 동시 열기 방식: 결론과 근거

**"단일 창 + 컨텍스트 루트 분리" 방식을 권장합니다.**
VS Code multi-root와 Obsidian 별도 창의 중간 지점으로, Baram만의 방식을 가져가는 게 좋습니다.

---

## 두 방식의 트레이드오프

### Option A: 별도 창 (Obsidian 방식)

```
창 1: work-vault          창 2: journal-vault
┌─────────────────┐       ┌─────────────────┐
│ 🌀 Baram        │       │ 🌀 Baram        │
│ [work-vault]    │       │ [journal]       │
│ ─────────────── │       │ ─────────────── │
│ 📁 project-a   │       │ 📔 2026-03-21   │
│ 📁 skills/     │       │ 📔 2026-03-20   │
└─────────────────┘       └─────────────────┘
```

**장점**

- 컨텍스트 완전 격리: 두 vault가 서로 간섭 없음
- OS 창 관리 (Mission Control, Alt+Tab)로 자연스럽게 전환
- Obsidian 사용자에게 친숙한 멘탈 모델

**단점**

- 창 간 파일 드래그&드롭 불가
- 두 vault의 내용을 동시에 참조할 수 없음 (Persona 1의 Skills 편집 시 치명적)
- 앱 인스턴스가 두 개이므로 메모리 2배 사용 (각 \~50MB → 100MB)
- Journal vault를 열면서 work vault의 파일을 계속 보고 싶을 때 불편

---

### Option B: 단일 창 multi-root (VS Code 방식)

```
┌──────────────────────────────────────────┐
│ 🌀 Baram                                 │
│ [work-vault] [journal] [oss-contrib] [+] │ ← 컨텍스트 탭
│ ─────────────────────────────────────── │
│ 사이드바                                  │
│ ▼ 📁 work-vault      ← 구분선으로 분리    │
│   ├── project-a.md                       │
│   └── skills/                            │
│ ▼ 📔 journal                             │
│   ├── 2026-03-21.md                      │
│   └── 2026-03-20.md                      │
│ ▼ 📁 oss-contrib                         │
│   └── CHANGELOG.md                       │
└──────────────────────────────────────────┘
```

**장점**

- 여러 vault의 파일을 동시에 탭으로 열어서 비교 가능
- 메모리 효율 (프로세스 1개)
- Skills 편집 시 work-vault와 journal을 동시에 참조 가능

**단점**

- 파일트리가 길어지면 인지 부하 증가
- vault 간 경계가 시각적으로 불분명해질 위험
- 링크/검색 scope가 어느 vault인지 혼란 가능

---

## Baram 권장 방식: "단일 창 + 컨텍스트 탭으로 명확히 구분"

두 방식의 장점만 취합니다.

### 핵심 설계

```
┌──────────────────────────────────────────────────────┐
│ 🌀 Baram   ● work-vault  ○ journal  ○ oss  [+]      │
│            ─────────────                             │
│            현재 활성 컨텍스트 = 하나                   │
├──────────────────────────────────────────────────────┤
│ 사이드바: 활성 컨텍스트의 파일트리만 표시               │
│                                                      │
│ ▼ 📁 work-vault  (현재 활성)                          │
│   ├── project-a.md                                   │
│   └── skills/                                        │
│                                                      │
│ ─────────────────────────────────────────────────    │
│ 에디터 탭: 컨텍스트 구분 없이 모든 열린 파일            │
│ [project-a.md] [2026-03-21.md ●] [skill-x.md]       │
│  (work)          (journal)        (oss)              │
│  ↑ 탭에 컨텍스트 색상 dot으로 출처 구분               │
└──────────────────────────────────────────────────────┘
```

**핵심 규칙:**

```
사이드바 파일트리 = 활성 컨텍스트 하나만 표시
에디터 탭         = 모든 컨텍스트 파일 공존 가능
링크/검색 scope   = 활성 컨텍스트 기준 (명시적 전환 가능)
```

이 규칙이 핵심입니다. 파일트리를 한 번에 다 보여주는 VS Code 방식의 인지 부하 문제를 피하면서, 파일은 컨텍스트 경계를 넘어 탭에서 공존할 수 있게 합니다.

---

### 컨텍스트 탭 UX 상세

```
컨텍스트 탭 클릭 → 파일트리가 해당 vault로 교체
                  (탭에 열린 파일들은 그대로 유지)

컨텍스트 탭 우클릭 메뉴:
  ├── 새 창으로 분리         ← Obsidian 방식으로 탈출 가능
  ├── 이 컨텍스트만 남기기
  ├── 설정 열기
  └── 닫기
```

"새 창으로 분리"를 제공함으로써 Obsidian 사용자도 원하는 대로 쓸 수 있습니다.

---

### 에디터 탭의 컨텍스트 표시

```
[● project-a.md]  [● 2026-03-21.md]  [● skill-x.md]
  파란 dot            초록 dot           주황 dot
  = work-vault        = journal          = oss-contrib
```

각 컨텍스트에 고유 색상을 부여 (`.baram/config.json`의 `color` 필드)하면 탭만 봐도 어느 vault 소속인지 즉각 인식됩니다.

## 페르소나별 검증

| 페르소나                    | 주요 시나리오                                      | 이 방식으로 해결되는가           |
| ----------------------- | -------------------------------------------- | ---------------------- |
| **Persona 1** (LLM 개발자) | skills-vault + journal-vault 동시 참조하며 개발일지 작성 | ✅ 두 파일을 탭에 나란히 열 수 있음  |

| **Persona 3** (연구자)     | 논문 vault + 메모 폴더 병행                          | ✅ 폴더를 두 번째 컨텍스트로 추가    |

## 별도 창 방식이 더 나은 경우

이 한 가지 상황에서는 별도 창이 우월합니다:

> **"두 vault의 문서를 화면에 나란히 놓고 동시에 편집하고 싶다"**

이 경우는 "새 창으로 분리" 기능으로 대응합니다. 기본은 단일 창이되, 필요 시 탈출구를 제공하는 전략입니다.

---

## 구현 복잡도 관점

```
별도 창 방식:
  → 창 간 통신 필요 (IPC 복잡도 증가)
  → 각 창이 독립 프로세스 or 같은 프로세스 내 멀티 윈도우 결정 필요
  → Tauri의 multi-window API 사용 (가능하지만 상태 동기화 까다로움)

단일 창 방식:
  → ContextManager 내에서 상태 관리 (앞서 설계한 구조 그대로)
  → 별도 창 분리는 나중에 추가 기능으로 제공
  → M1/M2 구현 부담 낮음
```

단일 창 방식이 **지금 당장 구현하기 훨씬 간단**하고, 나중에 별도 창 분리를 추가하는 건 가능하지만 반대는 어렵습니다. 즉, 단일 창으로 시작하는 게 리스크도 낮습니다.

## 최종 결론

| 결정                    | 이유                        |
| --------------------- | ------------------------- |
| **기본: 단일 창**          | 파일 간 참조, 메모리 효율, 구현 단순성   |
| **파일트리: 활성 컨텍스트 하나만** | 인지 부하 방지, 명확한 scope       |
| **에디터 탭: 컨텍스트 무관**    | 실제 사용 시 cross-vault 비교 가능 |
| **탈출구: 창 분리 옵션 제공**   | Obsidian 사용자 배려, 특수 사용 대응 |
| **M1에서는 단일 컨텍스트만**    | 복잡도 관리, 설계 검증 우선          |
